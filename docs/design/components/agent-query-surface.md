# Component Design: Agent Query Surface

## Summary
The agent-query-surface component adds the cache-first service query API and CLI query mapping that let agents perform source discovery, task lookup, handoff review, citation retrieval, recent inspection, link checking, stale-pointer inspection, sync-status checks, export, and diff without raw shell access or live network reads. The required functionality is not already present: the repository currently has only the entrypoint/CLI scaffold and no shared service query layer.

## Top-Level Alignment
This component sits at the approved `internal/service` boundary and is consumed by `internal/cli` and the first-slice `internal/mcp` read tools. It reuses `internal/cache.Store` and cache/index projections; it does not own SQLite schema, GitCode HTTP access, MCP transport internals, or derived-index production.

### Task 1: Add service query API
Outcome IDs: outcome-4, outcome-5, outcome-10
Outcome Role: primary_product
Decommission IDs: decommission-1, decommission-3
Change Type: add
Description: Add the `internal/service` query API that gives CLI and MCP one shared cache-first surface for source discovery, task lookup, handoff review, citation snippets, sync-status reads, and snapshot reads. The service also provides CLI-consumed DTOs for recent-change inspection, link checking, and stale-index reports by reading cache/index projections. Routine methods translate agent-facing operations into `internal/cache.Store` calls and never require a GitCode client for reads.
Existing Behavior / Reuse: `internal/service` is currently absent and the existing CLI scaffold does not implement real query behavior. Reuse the approved `internal/cache.Store` boundary for sources, identity map, links, chunks, sync status, snapshots, and derived index projections; do not duplicate schema ownership or direct SQL outside cache-owned interfaces. Keep GitCode adapter access out of all routine query paths.
Detailed Design: Add a `Service` type constructed with one required `internal/cache.Store` dependency and no required GitCode client for query reads. Add request/response DTOs for `SearchSourcesRequest`, `SearchSourceResult`, `GetSourceRequest`, `SourceRecord`, `ListSourcesRequest`, `SourceSummary`, `BacklinkResult`, `ResolveIDRequest`, `ResolvedID`, `SnippetRequest`, `SnippetResult`, `SyncStatusResult`, `RecentChangesRequest`, `RecentChangeResult`, `LinkCheckRequest`, `LinkCheckResult`, `BrokenLinkResult`, `StaleIndexRequest`, `StaleIndexResult`, `ExportSnapshotRequest`, `ExportSnapshotResult`, `DiffSnapshotRequest`, and `DiffSnapshotResult`. Implement `SearchSources`, `GetSource`, `ListSources`, `GetBacklinks`, `ResolveID`, `GetSnippet`, `GetSyncStatus`, `RecentChanges`, `LinkCheck`, `StaleIndex`, `ExportSnapshot`, and `DiffSnapshot` as cache-first methods that accept `context.Context`, validate arguments, resolve aliases through the identity map where applicable, delegate to `internal/cache.Store`, and normalize failures into typed service errors: `ErrNotFound`, `ErrCacheEmpty`, `ErrInvalidQuery`, `ErrRangeClamped`, `ErrStaleIndex`, and `ErrLinkCheckFailed`.

For the first-slice MCP-facing contract, this task supports exactly the approved eight MCP tools: `search_sources`, `get_source`, `list_sources`, `source_backlinks`, `resolve_id`, `sync_status`, `export_snapshot`, and `diff_snapshot`. `internal/mcp` maps those eight tools to the corresponding service methods and returns the service DTOs in `structuredContent`; `RecentChanges`, `LinkCheck`, and `StaleIndex` remain service DTOs and CLI-consumed methods only, with MCP exposure deferred unless the top-level architecture is revised. `search_sources` returns ordered records with `id`, `path`, `title`, `kind`, `status`, `snippet`, `line_start`, `line_end`, and `score`; `get_source` returns full `id`, `path`, `remote_alias`, `kind`, `title`, `body`, `status`, `labels`, `links`, and `backlinks`; `list_sources` returns paged source summaries; `source_backlinks` returns sources that link to the resolved target id; `resolve_id` returns `{id, path, remote_alias, kind, title}`; `sync_status` returns freshness fields; `export_snapshot` returns deterministic snapshot metadata/content; and `diff_snapshot` returns deterministic diff output plus changed source ids.

Search ordering is deterministic: descending FTS score when available, then ascending `updated_at`, then ascending stable `id`, then ascending `path`. Search snippets come from cached FTS snippet data when available; otherwise the service computes a line-bounded excerpt from the cached source body and reports `line_start`/`line_end` when line metadata is derivable. If line metadata is absent, human renderers receive `path:0:snippet` and JSON receives `line_start: null`, `line_end: null`; the service never reads the original file path to recover missing line data. `GetSnippet` extracts citation text from cached source body using requested line range or chunk byte offsets, clamps ranges beyond available content, returns a warning field when clamped, and treats invalid ranges as `ErrInvalidQuery`.

`RecentChanges` reads cache-maintained update metadata and sync/index projections only, sorts descending `updated_at` and then ascending stable `id`, and supports `kind`, `status`, `limit`, and `offset` filters. `LinkCheck` reads derived link projections produced by `internal/index` and returns broken or unresolved links without rebuilding indexes inline; suggested aliases come from identity-map lookups. `StaleIndex` reads index staleness projections and reports affected ids without modifying cache state; stale reports are query results, not runtime failures unless a caller requests strict behavior. Empty cache maps to `ErrCacheEmpty`, missing ids or aliases map to `ErrNotFound`, and all query methods complete with no GitCode client configured.

`ExportSnapshot` is deterministic service behavior over cache-only inputs. `ExportSnapshotResult` includes `snapshot_id`, `format`, `record_count`, `generated_at`, `content_hash`, `inline_content`, `output_path`, and `warnings`; records are ordered by ascending stable `id`, then `path`, links within a record are sorted by `target_id`, labels are sorted lexicographically, and timestamps are normalized to UTC RFC3339. Inline output is returned when no output path is requested or when rendered content is below the configured inline limit; otherwise `output_path` is returned and `inline_content` is empty. `DiffSnapshotResult` includes `base_snapshot_id`, `head_snapshot_id`, `format`, `changed_source_ids`, `added_source_ids`, `removed_source_ids`, `modified_source_ids`, `diff_text`, and `warnings`; all id lists are sorted ascending by stable id, and diff generation reads only stored snapshot/cache content.

Add CLI rendering for recent-change, link-check, and stale-index operations as Task 10 inspection commands. `recent --format json` returns an array of `RecentChangeResult` fields and text output renders `updated_at id path title`. `link-check --format json` returns `checked_count`, `broken_count`, `broken_links`, and `suggested_aliases`; text output renders one broken link per line and exits non-zero only in `--strict` mode when `broken_count > 0`. `stale-index --format json` returns `stale_count`, `affected_source_ids`, `missing_target_ids`, and `last_indexed_at`; text output summarizes counts and affected ids, and exits non-zero only in `--strict` mode when stale items exist. Index production remains owned by `internal/index`; service methods provide query DTOs from cached projections, and CLI only dispatches and renders those cache-backed query results.

Add CLI help text that states the exact mapping: `find -> list_sources`, `rg -n -> search_sources`, `rg --files -> list_sources`, `sed -n -> get_snippet`, handoff/review inspection -> `recent`, broken pointer search -> `link-check`, and stale derived data search -> `stale-index`. The minimum replacement sequence in help and tests is `ingest -> search_sources -> list_sources -> get_source -> source_backlinks -> sync_status`; `recent`, `link-check`, and `stale-index` are additional cache-backed inspection commands in the same query surface. Enforce the decommission invariant by keeping shell behavior out of query commands: handlers call `internal/service.Service` only and do not invoke `find`, `rg`, `sed`, perform direct markdown scans, call `internal/index` build routines, or call the GitCode adapter.

Define CLI-visible error behavior for all query commands. `ErrCacheEmpty` writes a concise message to stderr, emits no JSON records, and exits `2`; `ErrNotFound` exits `3`; `ErrInvalidQuery` exits `4`; clamped snippets exit `0` with the snippet on stdout and a warning on stderr or a JSON `warnings` field; non-strict stale-index and link-check findings exit `0` with findings in stdout; strict stale-index/link-check exits `5`; unexpected service errors exit `1`. JSON mode returns valid JSON for successful and warning-bearing results, while typed failures write machine-readable error JSON to stderr when `--format json` is requested.

## Cross-Cutting Constraints
- Routine agent reads must be cache-first and must not access the GitCode adapter or live network — this preserves the approved offline-read invariant for search, get, backlinks, snippets, sync status, recent, link-check, stale-index, export, and diff
- First-slice MCP `tools/list` remains exactly the approved eight tools — this preserves outcome-4 and prevents this component from expanding MCP scope beyond the approved architecture
- CLI and MCP query behavior must share the same service methods and DTOs where MCP exposure is approved — this keeps tool output behavior identical across human CLI use and MCP `tools/call` handlers
- The service uses one `internal/cache.Store` boundary — this follows the approved architecture and avoids redundant service-local persistence ports
- Stable source ids remain authoritative and remote ids are aliases — this keeps task lookup, backlink review, and citation retrieval consistent with the identity-map contract
- Shell replacement is semantic, not textual — output may be structured JSON or `path:line:snippet`, but it must provide equivalent discovery, lookup, citation, inspection, and review information without invoking `find`, `rg`, or `sed`

## Data And Control Flow
- Agent invokes a CLI query command — `internal/cli.Execute` parses command and flags — command handler creates a service request and selects text or JSON rendering
- MCP server receives `tools/call` for one of the approved eight tools — `internal/mcp` maps the tool name to the corresponding `internal/service` method — the returned DTO is emitted as structured tool output without redefining query behavior
- CLI calls `recent`, `link-check`, or `stale-index` — `internal/cli` calls service query methods — service reads cache/index projections and returns DTOs without triggering index rebuilds
- CLI or MCP calls the service query API — `internal/service.Service` validates request fields — service resolves ids and reads cache records, links, chunks, sync metadata, recent metadata, link-check projections, stale-index projections, snapshots, and diffs through `internal/cache.Store`
- Service returns typed query results — CLI renderer formats JSON or text and MCP returns the same DTO shape for approved tools in `structuredContent` — no query command reads source files directly or calls the GitCode adapter
- Backlink and snippet queries resolve stable ids first — identity map lookup precedes link or body/chunk retrieval — aliases and canonical ids produce equivalent query responses
- Link-check and stale-index queries read projections produced by `internal/index` — query methods do not rebuild indexes inline — stale or broken results are reported deterministically from cached projection state

## Component Interactions
- `internal/cli` -> `internal/service` — CLI query commands call `SearchSources`, `ListSources`, `GetSource`, `GetBacklinks`, `ResolveID`, `GetSnippet`, `GetSyncStatus`, `RecentChanges`, `LinkCheck`, `StaleIndex`, `ExportSnapshot`, and `DiffSnapshot` with parsed flags and render returned DTOs
- `internal/mcp` -> `internal/service` — MCP `tools/call` handlers for the approved first-slice tools `search_sources`, `get_source`, `list_sources`, `source_backlinks`, `resolve_id`, `sync_status`, `export_snapshot`, and `diff_snapshot` use the same service methods and DTOs as CLI query commands
- `internal/service` -> `internal/cache` — service query methods depend on the approved cache Store for source records, identity map resolution, links, chunks, sync metadata, derived index projections, snapshots, and diffs; service owns orchestration but not persistence
- `internal/index` -> `internal/cache` — index builds populate links, chunks, broken-link reports, and stale-index projections consumed by the agent query surface; query methods read these projections but do not rebuild them inline

## Rationale

Existing project behavior does not already provide this component-local surface, so the work is additive and directly traceable to the approved architecture’s shared service layer and shell-replacement outcomes.
