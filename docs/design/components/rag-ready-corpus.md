# Component Design: RAG-Ready Corpus Boundary

## Summary
The RAG-ready corpus boundary defines durable chunk provenance and deterministic corpus snapshots for later semantic retrieval. It stores citable chunk state now while keeping embeddings, vector storage, reranking, generated answers, chat memory, and model tuning deferred.

## Top-Level Alignment
This component implements the approved architecture’s Task 9 chunk/provenance model and contributes export/diff behavior from Tasks 5 and 14 through the shared service layer. It spans cache-owned chunk records, index-owned chunk generation, and service-owned snapshot APIs without adding live-network read paths or semantic retrieval runtime.

## Tasks

### Task 1: Add chunk provenance schema
Outcome IDs: outcome-9
Outcome Role: primary_product
Decommission IDs: none
Change Type: add
Description: Add the cache and indexing entities that make each source version addressable as deterministic, citable chunks. The chunk model is owned by the RAG-ready corpus boundary because future semantic retrieval must reuse the same byte offsets, heading paths, content hashes, aliases, and metadata without re-ingesting sources. This task stores the embedding slot now as nullable corpus state while leaving embedding generation out of runtime behavior.
Existing Behavior / Reuse: The current source contains only the CLI entrypoint and CLI scaffold; there is no cache package, index package, service package, chunk schema, chunker, or provenance model. Reuse the approved cache-first component split: cache owns durable records, index computes chunks, and service reads authoritative citations.
Detailed Design: Add a `Chunk` data structure with `ID`, `SourceID`, `ContentHash`, `ByteStart`, `ByteEnd`, `LineStart`, `LineEnd`, `HeadingPath`, `Text`, `NormalizedText`, `InheritedMetadata`, `OutboundLinks`, `ResolvedAliases`, and `Embedding` fields. Add cache store methods `UpsertChunk`, `GetChunks`, and schema migration support for a `chunks` table with `embedding BLOB DEFAULT NULL`; `source_id + content_hash` identifies the parent source version, while each chunk row is uniquely identified by deterministic `chunk_id = SHA-256(source_id + "\x00" + content_hash + "\x00" + byte_start)` and enforced by `PRIMARY KEY(id)` plus `UNIQUE(source_id, content_hash, byte_start)`, allowing multiple chunks per version.

Add an index chunker that accepts the exact UTF-8 markdown source bytes already stored in cache and computes `content_hash` over those original bytes. Byte ranges are zero-based half-open intervals `[byte_start, byte_end)` into the original byte slice; line ranges are one-based inclusive ranges `[line_start, line_end]` covering every source line touched by the byte range. The frontmatter provenance invariant is: when the source begins with `---\n` and has a valid closing `---` delimiter, no chunk byte range may include the frontmatter bytes, every emitted `Chunk.Text` must equal exactly `original_bytes[byte_start:byte_end]`, and parsed frontmatter key/value pairs are represented only in `InheritedMetadata`; if no closing delimiter exists, the leading text is treated as normal body text. This keeps `get_snippet(source_id, byte_start, byte_end)` and citation rendering compatible with chunk text because a citation snippet over the chunk range returns the same bytes as `Chunk.Text`.

Heading paths are built from ATX markdown headings only (`#` through `######`), trimming marker and surrounding whitespace; lower/equal heading levels replace the stack at that level, deeper levels append, and post-frontmatter pre-heading content uses an empty `HeadingPath`. Chunk boundaries are deterministic: start a new chunk at every ATX heading line and when adding the next paragraph/list/code-block line would exceed the configured soft limit of 4 KiB of source bytes; never split inside a fenced code block, and if a single block exceeds the soft limit it becomes one oversized chunk. Blank lines are retained in `Text` when they fall inside a chunk range. `NormalizedText` is produced from `Chunk.Text` by converting CRLF to LF, trimming leading/trailing Unicode whitespace, collapsing internal Unicode whitespace runs to one ASCII space outside fenced code blocks, and lowercasing ASCII letters only. Outbound links are extracted from markdown inline links and stable-id tokens, resolved aliases are filled from the cache identity map, unordered arrays are sorted lexicographically before persistence, and re-running the chunker over identical source bytes must produce byte-identical chunk rows and identical chunk ids.
Acceptance Criteria: A developer runs `go test ./internal/cache/... -run TestChunkSchemaEmbeddingColumn`; the cache API opens an in-memory SQLite store, inspects the `chunks` table, inserts multiple chunks sharing the same `source_id` and `content_hash` but different `byte_start`, observes `embedding` as NULL, and verifies duplicate `(source_id, content_hash, byte_start)` is rejected. A developer runs `go test ./internal/index/... -run TestChunkDeterminism`; the index runtime ingests markdown with frontmatter, pre-heading text, headings, links, and a fenced code block, then returns chunks with correct half-open byte offsets, inclusive line ranges, heading paths, inherited metadata, outbound links, resolved aliases, normalized text, and identical chunk ids on a second run over the same source. The same test verifies the frontmatter invariant by asserting no chunk range overlaps valid frontmatter bytes, `Chunk.Text == original_bytes[byte_start:byte_end]` for every chunk, frontmatter values appear only in `InheritedMetadata`, and a snippet/citation call over each chunk byte range returns the same text.

### Task 2: Add corpus snapshot APIs
Outcome IDs: outcome-9, outcome-14
Outcome Role: supporting_evidence
Decommission IDs: decommission-14
Change Type: add
Description: Add export and diff behavior for the RAG-ready corpus so sources, chunks, links, and provenance can be reviewed and consumed by later semantic retrieval pipelines. The export model belongs to this component because it defines the stable corpus boundary between cache-first GitCode MCP data and future retrieval systems. The task keeps `resolve_id`, `get_source`, `get_snippet`, backlinks, and sync status authoritative for citations rather than letting snapshot consumers invent citation state.
Existing Behavior / Reuse: Current CLI commands are stubs and there is no service export API, diff API, snapshot representation, or corpus export format. Reuse the approved service-layer ownership for `ExportSnapshot` and `DiffSnapshot`, and reuse chunk provenance from Task 1 as the snapshot’s corpus unit.
Detailed Design: Add a `Snapshot` model containing schema version, creation metadata, ordered source records, identity aliases, links/backlinks, sync freshness fields, and chunk records with provenance fields from `Chunk`. Add `ExportSnapshot(ctx, ExportSnapshotRequest) (ExportSnapshotResult, error)` where the request includes `Format` (`json`, `markdown`, or `sqlite`), optional `IncludeBody`, optional `SourceIDs`, and optional `OutputPath`; when `OutputPath` is empty the result returns inline bytes, otherwise it writes the snapshot and returns the path plus content hash. JSON output is the canonical deterministic machine corpus boundary for future RAG ingestion; markdown output is deterministic human review output. `Format:"sqlite"` is a logical snapshot/export convenience, not the canonical RAG boundary and not promised to be byte-identical as a SQLite database file; if implemented in the first slice, diff/import must compare its logical rows after loading rather than relying on SQLite file bytes.

Add `DiffSnapshot(ctx, DiffSnapshotRequest) (DiffSnapshotResult, error)` where each side is a `SnapshotRef` of kind `current`, `path`, or `bytes`; `current` means the service reads the current cache state, `path` reads a prior exported snapshot, and `bytes` is an already-loaded serialized snapshot. The diff output reports structured categories: `added_sources`, `removed_sources`, `changed_sources`, `added_chunks`, `removed_chunks`, `changed_chunks`, `added_links`, `removed_links`, `changed_aliases`, and `changed_sync_status`. Sources are compared by stable source id, chunks by deterministic chunk id, aliases by `(source_id, alias_kind, alias_value)`, and changed records include before/after content hashes plus changed field names.

Deterministic serialization is mandatory for JSON and markdown: sort sources by id, chunks by `(source_id, content_hash, byte_start, id)`, links by `(source_id, target_id, link_type)`, aliases by `(source_id, alias_kind, alias_value)`, and sync records by source id. Nested JSON metadata (`inherited_metadata`, `outbound_links`, `resolved_aliases`, labels, and any alias maps) is canonicalized recursively with stable object key ordering; arrays that are not semantically ordered, such as labels, aliases, outbound links, and resolved alias lists, are sorted lexicographically, while semantically ordered arrays such as `heading_path` preserve source order. Timestamps are emitted in UTC RFC3339Nano, zero values are omitted consistently, map iteration is never used directly for output ordering, and repeated JSON/markdown exports of identical cache state must be byte-identical.

For `decommission-14`, replace Day 7 corpus inspection and export/diff workflows with cache-derived snapshots only. The negative invariant is that `ExportSnapshot`, `DiffSnapshot`, and Day 7 corpus inspection do not read legacy markdown indexes or source files directly after ingest/index; they read only cache tables and previously exported snapshot bytes. CLI command handlers for `export` and `diff` call these service methods directly and write machine JSON or human markdown without requiring MCP.
Acceptance Criteria: A developer runs `go test ./internal/service/... -run TestExportDeterminism`; the service API populates cache state with nested metadata and aliases, exports twice through `ExportSnapshot(ctx, {Format:"json"})`, exports twice through `ExportSnapshot(ctx, {Format:"markdown"})`, and receives byte-identical output within each format. A developer runs `go test ./internal/service/... -run TestExportIncludesChunkProvenance`; the JSON snapshot contains source ids plus chunk `id`, `byte_start`, `byte_end`, `line_start`, `line_end`, `heading_path`, `content_hash`, `inherited_metadata`, `outbound_links`, and `resolved_aliases`. A developer runs `go test ./internal/service/... -run TestDiffSnapshot`; the service compares `base` from exported JSON bytes to `head` from current cache state and reports added/removed/changed source and chunk categories with changed field names. Running `gitcode-mcp export --format json` after fixture ingest returns valid deterministic JSON suitable for the Day 7 offline walkthrough, and the test verifies no legacy markdown index or source file read occurs during export/diff after prior ingest/index. If `gitcode-mcp export --format sqlite` is exposed, a service test verifies it can be loaded and logically diffed against the same current cache state, without asserting byte-identical SQLite file output.

## Cross-Cutting Constraints
- Cache remains the authoritative corpus source — future semantic retrieval may propose chunks, but citation and freshness must continue through cache-backed ids, snippets, backlinks, and sync status
- Snapshot output must be deterministic for canonical JSON and markdown — repeated exports of unchanged cache state must be byte-identical for review, audit, and RAG corpus ingestion, including nested metadata and alias structures
- SQLite snapshot bytes are not the canonical deterministic corpus boundary — SQLite exports, if present, are compared as logical snapshots to avoid untestable binary-stability promises
- Embedding runtime is deferred — the schema may contain a nullable embedding slot, but no provider calls, vector store, reranking, answer generation, chat memory, or model tuning are added in this slice
- Chunk identity is version-scoped — `source_id + content_hash` identifies the parent source version, and `chunk_id` or `(source_id, content_hash, byte_start)` identifies each row within that version
- Chunk text is byte-citable — for every chunk, `Text` equals the original source byte slice for `[byte_start, byte_end)`, and valid frontmatter is excluded from chunk ranges while represented in inherited metadata

## Data And Control Flow
- Source records enter cache through ingest or sync — index chunker reads exact cached source bytes and writes deterministic chunk rows — cache owns persisted provenance
- Valid frontmatter is parsed before chunking — chunk ranges start after the frontmatter block — snippets over chunk byte ranges return exactly `Chunk.Text`
- Service export reads sources, identity aliases, links, freshness, and chunks from cache — canonical JSON/markdown serializer orders and normalizes records — CLI or MCP receives deterministic snapshot content
- Snapshot diff receives two stable snapshot refs — service compares records by stable ids and chunk ids — caller receives structured added, removed, and changed corpus entries
- Day 7 corpus inspection uses prior ingest/index state — export/diff operate only on cache tables or exported snapshot bytes — legacy markdown indexes are not read in product runtime

## Component Interactions
- `internal/index` -> `internal/cache` — index computes `Chunk` rows and persists them only through cache store methods, preserving cache as sole database writer
- `internal/service` -> `internal/cache` — `ExportSnapshot` and `DiffSnapshot` read cache-owned source, alias, link, freshness, and chunk records without live GitCode calls
- `internal/cli` -> `internal/service` — CLI `export` and `diff` invoke the same service snapshot APIs that MCP can expose later, keeping transport behavior equivalent
- `future RAG layer` -> `rag-ready-corpus` — semantic retrieval consumes canonical JSON chunks or cache chunk rows as candidates, then resolves citations through stable cache ids and snippet ranges

## Rationale
The approved architecture materially affects the RAG-ready corpus boundary because it requires chunk provenance and deterministic exports before embeddings or vector search exist. These tasks are component-local: they define the corpus data contract and export semantics that later retrieval systems depend on while preserving the first slice’s cache-first, read-first behavior.
