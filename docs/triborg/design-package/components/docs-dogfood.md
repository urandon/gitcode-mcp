# Component Design: Docs & Dogfood Feedback

## Summary
Docs & Dogfood Feedback is affected because the approved architecture assigns this component the public-safe documentation package, executable dogfood checklist, fixture/live validation guidance, and sanitized feedback gate. The component documents and validates product surfaces owned elsewhere; it does not implement runtime CLI, MCP, cache, adapter, credential, write, or snapshot behavior.

## Top-Level Alignment
This component turns the minimum dogfood slice into durable, executable guidance: install/configure, bind a repository, sync/index fixtures, read through CLI/MCP, validate fixture/live behavior, and record public-safe feedback. Its outputs are docs, smoke/check scripts, redacted transcripts, checklist evidence, and validator artifacts traceable to Tasks 13-16.

### Task 1: Docs Smoke Package Add
Outcome IDs: outcome-14
Outcome Role: primary_product
Decommission IDs: none
Change Type: add
Description: Add the public-safe documentation package and docs smoke product check for the minimum dogfood workflow. The package owns command walkthroughs and expected diagnostics, while runtime behavior remains owned by config, repo binding, cache, MCP, write, and adapter components. The smoke check proves documented fixture-mode commands either succeed or return their documented diagnostic without exposing secrets or private paths, and it includes a fixture-backed MCP read rather than readiness-only validation.
Existing Behavior / Reuse: Reuse existing `README.md`, `docs/architecture.md`, `docs/cache-and-sync-model.md`, and `docs/gitcode-api-discovery.md` as source material. Confirmed absent are a complete quickstart, config/secrets reference, repo-binding guide, CLI/MCP walkthroughs, write walkthrough, troubleshooting/doctor guide, fixture-capture guide, executable docs smoke runner, and shared dogfood transcript safety policy. `docs/dogfood-checklist.md` is intentionally not added by this task; it is referenced as a later checklist dependency owned only by Task 3.
Detailed Design: Add the documentation set under `docs/`: `install.md`, `config-reference.md`, `repo-binding.md`, `secrets.md`, `mcp-setup.md`, `read-walkthrough.md`, `write-walkthrough.md`, `troubleshooting.md`, and `fixture-capture.md`. Add a shared dogfood safety/redaction policy as `project/dogfood/lib/safety.sh` plus `project/dogfood/fixture-allowlist.txt`; this policy is reused by `docs-smoke.sh`, `validate-fixtures.sh`, `the run.sh`, and `check-dogfood-feedback`. Keep the shared policy minimal: it exposes only `load_fixture_allowlist`, `redact_transcript`, `assert_public_safe_transcript`, and `write_redacted_transcript`. These functions load fixture ids/placeholders, replace temp home/cache paths with `<REDACTED_PATH>`, tokens with `<REDACTED_TOKEN>`, auth/cookie headers with `<REDACTED_SECRET>`, live hostnames outside the fixture allowlist with `<REDACTED_HOST>`, raw live responses with `<REDACTED_RESPONSE>`, fail on unredacted public-safety violations, and write the sanitized transcript.

Add `project/dogfood/docs-smoke.sh`, invoked as `project/dogfood/docs-smoke.sh --fixture-config testdata/configs/dogfood.yaml --cache-path <tmp>/gitcode-mcp.db --transcript <tmp>/docs-smoke.md [--live]`. The runner uses a curated command manifest embedded in the script or stored as `project/dogfood/docs-smoke.commands`; each entry has `id`, `doc_path`, `command`, `allowed_outcomes` (`success`, `documented_diagnostic`, `live_skip`), `requires_network`, `fixture_only`, and `redaction_profile`. The fixture smoke invariant is concrete: fixture steps run only through the fixture adapter and cache-only runtime surfaces owned by `gitcode_adapter` and `cache_sync`; routine CLI/MCP reads after fixture sync must use the fixture cache and must not attempt live access. Any command marked `requires_network=true` is classified as `live_skip` unless `--live`, `GITCODE_LIVE_TEST=1`, and token presence all pass. The runner exits nonzero if a `fixture_only=true` step attempts live access, if a live/network-required step runs without all gates, if a documented command has an undocumented outcome, or if transcript safety fails.

The MCP smoke portion starts the documented MCP mode against the fixture cache and performs an actual read. The preferred route is HTTP/SSE: start `gitcode-mcp mcp serve --transport http-sse --bind 127.0.0.1:<port> --cache-path <tmp>/gitcode-mcp.db`, then POST one JSON-RPC `tools/call` request to `/message` for `get_snippet` using the fixture `repo_id`, source type, record id, offset, and length from `project/dogfood/fixture-allowlist.txt`; verify the response contains the documented fixture snippet or documented diagnostic. If the documented smoke profile uses stdio, invoke the same tool through the documented stdio client harness and compare the response to the documented CLI `get-snippet` output for the same fixture cache. Add script units named `load_manifest`, `prepare_fixture_env`, `run_step`, `run_mcp_read_step`, `classify_outcome`, and `write_transcript`; all transcript writing delegates to `project/dogfood/lib/safety.sh`.

The write walkthrough is validated in fixture mode without requiring live mutation. Manifest entries for write docs must be classified as one of: `dry_run` product path for commands that can run with `--dry-run`, `documented_diagnostic` for unavailable write components/adapters, or `manifest_excluded_until_cli_write` with a documented reason until the write component exists. No write walkthrough entry may require live mutation in the default fixture smoke.
Acceptance Criteria: A new developer runs `project/dogfood/docs-smoke.sh --fixture-config testdata/configs/dogfood.yaml --cache-path <tmp>/gitcode-mcp.db --transcript <tmp>/docs-smoke.md` against sanitized fixtures. The target product surface is the documented CLI/MCP workflow from install through config, repo binding, fixture adapter sync/index, offline cache-only CLI read, MCP HTTP/SSE `/message` or documented stdio `tools/call` read, fixture validation, dry-run or documented-diagnostic write walkthrough behavior, and documented live-skip behavior. The visible outcome is a redacted transcript where every curated documented command succeeds, returns the documented diagnostic, or is skipped as credential-gated live validation; fixture-only steps fail if they attempt live access; the MCP read step returns the documented fixture snippet or documented read diagnostic for the same cache. Executable evidence is the local `project/dogfood/docs-smoke.sh` command; `go test ./...` is not claimed to exercise this shell smoke unless a later explicit Go wrapper is added by an owning task.

### Task 2: Fixture Validation Gate Add
Outcome IDs: outcome-13
Outcome Role: supporting_evidence
Decommission IDs: none
Change Type: validate
Description: Add the docs-owned validation gate for fixture and optional live-validation behavior. This task supports API discovery evidence by documenting how maintainers run offline fixture validation and credential-gated live checks safely. It does not implement the GitCode adapter; it verifies that docs and durable artifacts describe fixture/live behavior with shared redaction and skip semantics.
Existing Behavior / Reuse: Reuse `docs/gitcode-api-discovery.md` evidence rules: no credentials, sanitized fixtures, API version/date/scope notes, and separation of documented versus reverse-engineered behavior. Reuse the shared dogfood safety/redaction policy from `project/dogfood/lib/safety.sh` and the fixture identifiers from `project/dogfood/fixture-allowlist.txt`; existing docs do not yet provide a runnable fixture/live validation gate, skip contract, or durable redaction requirement tied to dogfood smoke output.
Detailed Design: Extend `docs/fixture-capture.md` with fixture validation commands, live-validation preconditions, redaction rules, fixture allowlist maintenance, and durable artifact policy. Add `project/dogfood/validate-fixtures.sh`, invoked as `project/dogfood/validate-fixtures.sh --fixtures testdata/fixtures --transcript <tmp>/fixture-validation.md [--live]`. The script runs offline fixture validation first, records whether live validation is skipped, and delegates runtime API behavior to adapter-owned tests such as `go test ./...` and optional `GITCODE_LIVE_TEST=1 go test -run Live` without embedding adapter logic.

The gate classifies results as `offline_pass`, `offline_fail`, `live_skipped_no_flag`, `live_skipped_no_token`, `live_pass_redacted`, or `live_fail_redacted`. Classification is script-local; only allowlist loading, redaction, public-safety assertion, and redacted transcript writing are delegated to `project/dogfood/lib/safety.sh`. The invariant is that fixture validation must pass without credentials and live validation must never run unless explicitly enabled and token presence is detected without printing the token.
Acceptance Criteria: A maintainer runs `project/dogfood/validate-fixtures.sh --fixtures testdata/fixtures --transcript <tmp>/fixture-validation.md` without credentials. The target product surface is fixture/live validation documentation and local validation command behavior. The visible outcome is offline fixture validation passing or producing documented diagnostics, live validation reported as skipped, and a redacted transcript with no secrets/private paths/raw responses according to the shared safety policy. Executable evidence is the local validation command and optional `GITCODE_LIVE_TEST=1` run showing redacted live output when credentials are intentionally provided.
Workload: 0.6 MM

### Task 3: Dogfood Checklist Add
Outcome IDs: outcome-15
Outcome Role: primary_product
Decommission IDs: none
Change Type: add
Description: Add the one-week dogfood implementation checklist as the component-owned product artifact. The checklist orders the approved architecture into executable day slices and stores redacted evidence without overwriting prior runs. It culminates in offline CLI and MCP reads for one fixture issue and one fixture wiki page.
Existing Behavior / Reuse: Reuse the approved milestone ordering: Day 1 config/repo, Day 2 fixture sync/index, Day 3 CLI reads, Day 4 MCP parity/transport, Day 5 concurrency/write safety, Day 6 snapshot integrity, Day 7 docs/live validation/feedback. Reuse docs-smoke command IDs and the shared dogfood safety/redaction policy from `project/dogfood/lib/safety.sh`; existing backlog and architecture notes contain ordering but no executable checklist artifact, evidence storage model, or replacement-command rule.
Detailed Design: Add `docs/dogfood-checklist.md` as the human-readable checklist and `project/dogfood/checklist.md` as the append-only evidence log template. Each slice contains `slice_id`, `required_prior_slices`, `command`, `expected_fixture_result`, `actual_redacted_result`, `transcript_path`, `status`, `blocker`, and `next_action`. Store transcripts under `project/dogfood/evidence/` with sanitized names such as `day1-config-repo.md`; generated or local-only transcripts may be gitignored while the template and sanitized examples remain repo-owned.

Add `project/dogfood/the run.sh`, invoked as `project/dogfood/the run.sh --slice dayN --cache-path <tmp>/gitcode-mcp.db --transcript project/dogfood/evidence/dayN.md`. The runner reuses docs-smoke command IDs where possible and delegates allowlist loading, transcript redaction, public-safety assertion, and redacted transcript writing to `project/dogfood/lib/safety.sh`. Replacement commands are recorded by appending a new evidence entry with `replaces_command_id`, `reason`, and `supersedes_transcript`; prior evidence is never edited or deleted. A slice can be marked complete only when its product check passes and all previous slices either remain runnable or have a documented replacement command with preserved prior evidence.
Acceptance Criteria: A follow-up implementer runs `project/dogfood/the run.sh --slice day1` through `--slice day7` against sanitized fixtures. The target product surface is the ordered dogfood checklist and its local command evidence. The visible outcome is per-slice pass/fail diagnostics with redacted transcripts, preserved replacement-command history when a command changes, and final evidence showing offline CLI and MCP reads for one fixture issue and one fixture wiki page. Executable evidence is the local checklist command sequence without live credentials.

### Task 4: Feedback Validator Add
Outcome IDs: outcome-16
Outcome Role: primary_product
Decommission IDs: none
Change Type: add
Description: Add the public-safe dogfood feedback artifact and architecture-aligned `check-dogfood-feedback` validator. The artifact captures runtime gap evidence, MCP/install friction, missing metadata, and design-agent prompt/config improvements after the dogfood slice runs. The validator enforces the repository boundary with deterministic line-referenced findings and bounded fixture allowlist rules.
Existing Behavior / Reuse: Reuse AGENTS boundary rules, `docs/gitcode-api-discovery.md` evidence rules, the shared fixture allowlist from `project/dogfood/fixture-allowlist.txt`, and the safety patterns in `project/dogfood/lib/safety.sh` as policy inputs. Existing project task and handoff conventions do not include `project/dogfood/feedback.md`, seeded validator fixtures, or the stable public-safety command `project/dogfood/check-dogfood-feedback`.
Detailed Design: Add `project/dogfood/feedback.md` with structured sections: run context, completed checklist slice, commands/transcripts referenced, observed friction, missing metadata, MCP client setup friction, prompt/config improvement, follow-up owner, and redaction checklist. Add `project/dogfood/check-dogfood-feedback` as the stable validator command, invoked as `project/dogfood/check-dogfood-feedback project/dogfood/feedback.md`; it may delegate internally to a shell implementation, but the stable command name is `check-dogfood-feedback`. Add seeded cases under `project/dogfood/testdata/feedback-pass.md` and `project/dogfood/testdata/feedback-fail.md`.

The validator loads allowed fixture identifiers from `project/dogfood/fixture-allowlist.txt`, which contains the documented sample repo ids, fixture issue aliases, fixture wiki aliases, fixture hostnames, localhost values, and accepted placeholders: `<REDACTED_TOKEN>`, `<REDACTED_SECRET>`, `<REDACTED_PATH>`, `<REDACTED_HOST>`, `<REDACTED_RESPONSE>`, and `<FIXTURE_REPO>`. Fatal deny patterns are bounded to bearer/API tokens, `Authorization:` headers, `Cookie:`/`Set-Cookie:`, token-like high-entropy strings, absolute user paths such as `/Users/<name>/`, `/home/<name>/`, Windows profile paths, private/internal hostnames, raw JSON response blocks lacking `<REDACTED_RESPONSE>` or fixture markers, and explicit repo/tracker/wiki identifier tokens that match documented identifier syntaxes but are absent from the fixture allowlist. Unknown prose identifiers that do not match those syntaxes produce warnings only when the validator cannot classify them, and warnings are reported separately from fatal findings so false positives do not block sanitized feedback. Findings are emitted as `path:line:category:message` and the command exits nonzero on any fatal finding. The pass seed must include only placeholders and fixture identifiers; the fail seed must include at least one token, cookie, private path, private host, raw response block, and non-allowlisted repo/tracker/wiki identifier, each reported with a line reference.
Acceptance Criteria: A coordinator fills `project/dogfood/feedback.md` after running the dogfood checklist and runs `project/dogfood/check-dogfood-feedback project/dogfood/feedback.md`. The target product surface is the public-safe feedback artifact and stable validator command. The visible outcome is zero fatal findings for sanitized feedback, warnings only for bounded unknown-identifier cases, and categorized line-referenced failures for seeded token, cookie, private path, private host, raw-response, and non-allowlisted identifier examples. Executable evidence is `project/dogfood/check-dogfood-feedback` over both pass and fail fixtures.

## Cross-Cutting Constraints
- Public-safe artifacts only — docs, transcripts, fixture instructions, checklist evidence, and feedback must not expose credentials, cookies, private paths, internal URLs, private repository names, or unsanitized API responses
- Cache-first routine reads — documented CLI and MCP read flows must show offline fixture/cache reads after explicit sync and must not imply implicit live network reads
- Live validation is explicit and credential-gated — docs and smoke checks skip live behavior unless the user opts in and token presence is confirmed without printing token value
- Shared dogfood safety policy — `docs-smoke.sh`, `validate-fixtures.sh`, `the run.sh`, and `check-dogfood-feedback` reuse the same minimal allowlist/redaction/transcript-writing policy to avoid drift in public-safety behavior
- Evidence must be executable — docs package, checklist, fixture validation, and feedback validation must produce local command evidence rather than review-only artifacts

## Data And Control Flow
- Documentation author updates `docs/*.md` — `project/dogfood/docs-smoke.sh` executes curated documented fixture commands and one fixture-backed MCP read — undocumented outcomes become smoke failures until docs or product diagnostics are corrected
- Maintainer runs `project/dogfood/validate-fixtures.sh` — offline fixture validation runs first — optional live validation is skipped unless explicit env/flag/token gates pass and all durable output is redacted through the shared safety policy
- Implementer follows `docs/dogfood-checklist.md` — `project/dogfood/the run.sh` records append-only redacted evidence under `project/dogfood/evidence/` — replacement commands append superseding evidence without mutating prior entries
- Coordinator records `project/dogfood/feedback.md` — `project/dogfood/check-dogfood-feedback` scans with fixture allowlist and bounded deny rules — only zero-fatal-finding feedback is accepted as dogfood evidence

## Component Interactions
- `config_credential` -> `docs_dogfood` — owns `config init`, `config locate`, `config show --redacted`, `auth status`, and doctor diagnostics; docs_dogfood documents and smoke-validates expected outputs, and failures are handed back as docs/product-diagnostic mismatches, not fixed inside docs_dogfood
- `repo_binding` -> `docs_dogfood` — owns `repo add`, `repo status`, repo ids, aliases, and collision behavior; docs_dogfood consumes fixture repo identifiers from the fixture allowlist and documents expected scoped behavior
- `cache_sync` -> `docs_dogfood` — owns sync/index/cache status behavior and fixture cache creation; docs_dogfood documents command sequences and records smoke/checklist evidence against fixture adapter and cache-only runtime surfaces
- `mcp_server` -> `docs_dogfood` — owns stdio and HTTP/SSE runtime routes, health/readiness, and MCP tool responses; docs_dogfood documents client setup and validates one fixture-backed read tool invocation without implementing transport logic
- `gitcode_adapter` -> `docs_dogfood` — owns fixture/live adapter behavior, API errors, and redaction implementation for adapter artifacts; docs_dogfood documents fixture/live validation gates and rejects unsafe durable evidence
- `docs_dogfood` -> `all_runtime_components` — publishes public-safe smoke/checklist failures as evidence of documentation drift or product diagnostic mismatch; owning runtime components decide whether to change behavior or docs expectations

## Rationale
The approved architecture assigns public-safe docs, day-slice dogfood ordering, fixture/live validation guidance, and sanitized feedback directly to `docs_dogfood`. Current docs are foundational but incomplete for an executable minimum dogfood slice, so component-local docs, scripts, evidence templates, shared safety policy, MCP read smoke, and validators are required.
