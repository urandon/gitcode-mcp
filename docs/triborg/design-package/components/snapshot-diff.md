# Component Design: Snapshot & Diff

## Summary
Snapshot & Diff adds durable, repo-scoped stored snapshots for indexed cache state. It replaces transient current-state export/diff behavior with stored-ID resolution, citation-bearing exports, deterministic stored-snapshot diffs, explicit missing/stale-index warnings, and typed not-found errors.

## Top-Level Alignment
This component implements architecture component `a9` and resolves only Request Task 11 through `snapshot-diff-delta-1`. It owns `internal/snapshot` snapshot semantics, consumes `internal/index` chunk rows, persists through `internal/cache` `snapshots` and `snapshot_chunks` transactions, and is invoked/rendered by `internal/cli` export/diff commands.

## Tasks

### Task 1: SnapshotStore stored diff
Outcome IDs: outcome-11
Outcome Role: primary_product
Decommission IDs: decommission-3, decommission-8
Change Type: add
Description: Add the `SnapshotStore` product contract for repo-scoped snapshot creation, export, and diff over stored rows. The component owns durable snapshot identity, citation-range assembly, missing/stale-index warning policy, and strict stored-ID lookup for `index`, `export-snapshot`, and `diff-snapshot`. Diff results must be derived only from the two resolved stored snapshots and must never consult current cache records after ID resolution.
Existing Behavior / Reuse: The repository has a CLI entrypoint under `cmd/gitcode-mcp` and the approved layout reserves `internal/cli`, `internal/cache`, `internal/index`, and `internal/snapshot` for this work. Existing scaffold-level command/service concepts for snapshot export or diff may be reused only as invocation or rendering shells if present; durable `SnapshotStore` semantics, `snapshots` / `snapshot_chunks` persistence, stored-ID-only resolution, and not-found-on-unknown product behavior are confirmed as new component work. Any current product fallback that compares current/current when explicit snapshot IDs are unresolved is disabled for `diff-snapshot`; path/bytes parsing or transient diff helpers, if useful, may remain internal test utilities but not product runtime behavior.
Detailed Design: Add `internal/snapshot` with a `SnapshotStore` interface: `CreateSnapshot(ctx, req) (StoredSnapshot, error)`, `GetSnapshot(ctx, repoID, snapshotID string) (StoredSnapshot, error)`, `ExportSnapshot(ctx, repoID, snapshotID string, format ExportFormat) (SnapshotExport, error)`, and `DiffSnapshots(ctx, repoID, baseID, headID string, format DiffFormat) (DiffSnapshotResult, error)`. `internal/cli` invokes these methods for `index` snapshot creation, `export-snapshot --repo --id`, and `diff-snapshot --repo --base-id --head-id`; CLI rendering consumes only `SnapshotExport`, `DiffSnapshotResult`, typed `not_found`, and typed consistency errors. `internal/cache` owns the SQLite transactions for `snapshots` and `snapshot_chunks`; `internal/index` owns source chunk rows and provides deterministic indexed chunk data consumed during `CreateSnapshot`.

Add data classes in `internal/snapshot`: `StoredSnapshot`, `StoredSnapshotChunk`, `SnapshotWarning`, `SnapshotExport`, `DiffSnapshotResult`, `DiffEntry`, and `SnapshotConsistencyError`. `StoredSnapshot` contains `RepoID`, `SnapshotID`, `CreatedAt`, `SchemaVersion`, `ManifestHash`, `ChunkSetHash`, `ChunkCount`, `Warnings`, and minimal immutable manifest sections needed for non-chunk export/diff: source records summary, identity/link summary, sync summary, and warning list. `StoredSnapshotChunk` contains `RepoID`, `SnapshotID`, `ChunkID`, `SourceType`, `SourceID`, `RecordID`, `SourceContentHash`, `SourceRevisionHash`, `IndexBuildID`, `ChunkContentHash`, `ByteStart`, `ByteEnd`, `LineStart`, `LineEnd`, `HeadingPath`, `Ordinal`, `Text`, and citation availability metadata. Citation ranges are durably represented only by `snapshot_chunks` range fields; if ranges are missing, non-positive, or outside the source bounds at creation time, `CreateSnapshot` stores the chunk and adds a `missing_citation` `SnapshotWarning` with affected `chunk_ids`.

The authoritative persistence model is normalized and bounded: `snapshots` is authoritative for identity, manifest sections, warning list, `chunk_count`, `manifest_hash`, and `chunk_set_hash`; `snapshot_chunks` is authoritative for chunk membership, chunk text, citation ranges, ordering, and chunk-level hashes. Export and diff reconstruct canonical payloads from the requested stored `snapshots` row plus its stored `snapshot_chunks` rows only. After `GetSnapshot` resolves `(repo_id, snapshot_id)`, `ExportSnapshot` and `DiffSnapshots` must not rebuild chunks, scan current records, query current index rows, or compare against current sync state; consistency checks are bounded to counting stored chunk rows and recomputing hashes over the stored rows for the resolved snapshot IDs.

Define canonical hash inputs explicitly. `ManifestHash` is SHA-256 over canonical JSON of the stored manifest sections: `repo_id`, `snapshot_id`, `schema_version`, normalized source summary entries ordered by `(source_type, source_id, record_id)`, normalized identity/link/sync summary entries, and normalized warnings ordered by `(kind, source_id, chunk_id, detail_code)`, excluding `created_at` and renderer-only formatting. `ChunkSetHash` is SHA-256 over canonical JSON lines of stored chunk rows ordered by stable chunk key `(source_type, source_id, record_id, ordinal, chunk_id)`, including `chunk_id`, `source_type`, `source_id`, `record_id`, `source_content_hash`, `source_revision_hash`, `index_build_id`, `chunk_content_hash`, byte/line ranges, `heading_path`, `ordinal`, and `text_hash`; it does not include full chunk text when `chunk_content_hash` is present. Creation writes the `snapshots` row and all `snapshot_chunks` rows in one `internal/cache` transaction; export/diff return a typed consistency error if stored row count or recomputed stored-row hash differs from `snapshots.chunk_count` or `snapshots.chunk_set_hash`.

`CreateSnapshot` runs immediately after indexing for the same repo or when explicitly requested by `index`. It reads repo-filtered source records and sync metadata through `internal/cache`, reads current deterministic chunk rows through `internal/index`, and stores immutable snapshot rows in one transaction. For each source record, if no indexed chunks exist, it stores a `missing_index` warning; if chunk metadata references an older `SourceContentHash`, `SourceRevisionHash`, or `IndexBuildID` than the source/sync state at creation time, it stores a `stale_index` warning. These warnings are persisted in the `snapshots` manifest and are later exported or diffed as stored data; warnings are not recalculated during export or diff.

`ExportSnapshot` first resolves `(repo_id, snapshot_id)` and validates stored consistency. It renders deterministic JSON or Markdown with stable source ordering `(source_type, source_id, record_id)`, stable chunk ordering `(source_type, source_id, record_id, ordinal, chunk_id)`, and warning ordering `(kind, source_id, chunk_id, detail_code)`. Unknown IDs return typed not-found before rendering and before any current-state read. Export always includes the requested `repo_id`, `snapshot_id`, stored chunks, byte/line citation ranges, and stored warnings for `missing_index`, `stale_index`, or `missing_citation`.

`DiffSnapshots` resolves both stored IDs under the same requested `repo_id`, validates each stored snapshot independently, then compares only the two stored canonical payloads. Stable comparison keys are: sources by `(source_type, source_id, record_id)`, chunks by `(source_type, source_id, record_id, ordinal, chunk_id)`, warnings by `(kind, source_id, chunk_id, detail_code)`, links by their normalized stored link key, and identity aliases by normalized alias key. Diff output categories are `added`, `removed`, `modified`, and `unchanged`; entries are ordered first by category in that order, then by entity kind `source`, `chunk`, `warning`, `link`, `identity`, then by the stable comparison key. Compared fields for chunks are `chunk_content_hash`, `source_content_hash`, `source_revision_hash`, `index_build_id`, byte/line citation ranges, `heading_path`, and `text_hash`; compared fields for sources and manifest entities are their stored hashes and normalized scalar fields. If only citation byte/line ranges change and chunk text/hash is unchanged, the chunk diff entry is `modified` with reason `citation_range_changed`; if only warnings change, the warning entries are added/removed and the parent source/chunk is not marked modified unless its own compared fields changed. Unchanged entries may be omitted by default in CLI output but are available in JSON when `--include-unchanged` is requested.

`DiffSnapshots` enforces `decommission-3` by returning typed not-found naming `base_id` or `head_id` if either stored ID is absent and by never substituting the current cache snapshot. `ExportSnapshot` enforces `decommission-8` by rendering stored `missing_index`, `stale_index`, and `missing_citation` warnings instead of silently omitting incomplete chunks/citations. Product runtime keeps any legacy current/current diff path inaccessible from `diff-snapshot --base-id --head-id`; if retained, it is limited to internal tests or explicitly named non-product helpers.
Acceptance Criteria: A developer runs `gitcode-mcp index --repo fixture-repo`, captures a stored snapshot ID, then runs `gitcode-mcp export-snapshot --repo fixture-repo --id <id> --format json`; the CLI response includes the requested `repo_id`, `snapshot_id`, repo-scoped source summaries, stored chunks from `snapshot_chunks`, byte/line citation ranges, and deterministic machine-readable warnings, with executable evidence from a CLI integration test over fixture snapshots. A fixture test mutates one cached record after indexing without rebuilding chunks, then creates and exports a snapshot; `export-snapshot` reports stored `stale_index` for the affected source, while a fixture source with no chunks reports stored `missing_index`, with evidence from the same CLI test suite. A developer runs `gitcode-mcp diff-snapshot --repo fixture-repo --base-id <valid-base> --head-id <valid-head> --format json`; the CLI returns deterministic `added`, `removed`, and `modified` entries ordered by the stored comparison contract, including `citation_range_changed` for range-only chunk changes, and service/CLI tests prove current cache mutations after snapshot creation do not affect the diff. A developer runs `gitcode-mcp diff-snapshot --repo fixture-repo --base-id missing --head-id <valid-head>`; the CLI returns typed not-found for `base_id` and does not produce `changed:false` or a current/current diff, with executable evidence from CLI and `internal/snapshot` tests.

## Cross-Cutting Constraints
- Repo scope is mandatory for every stored snapshot and chunk row — snapshot IDs resolve only under the requested `repo_id` because all cache/read/export operations are repo bounded
- Snapshot export and diff are deterministic — reviewable artifacts require stable ordering, canonical hash inputs, stored IDs, and stored chunk/citation rows rather than mutable current cache state
- Missing or stale index state is visible — export responses include machine-readable stored warnings so CLI/MCP users do not mistake partial artifacts for complete snapshots
- Stored snapshot consistency is bounded and local — `snapshots.chunk_count`, `snapshots.chunk_set_hash`, and `snapshots.manifest_hash` are checked against stored `snapshot_chunks` rows only, never by rescanning current cache records
- Unknown snapshot IDs are truthful errors — `diff-snapshot` must return not-found for missing base/head IDs and must not compare current state to itself

## Data And Control Flow
- `index` creates deterministic chunk rows, then `SnapshotStore.CreateSnapshot` persists a stored snapshot — `internal/index`, `internal/snapshot`, `internal/cache` — creation reads current repo-scoped indexed state once and writes `snapshots` plus `snapshot_chunks` in one transaction
- `export-snapshot` resolves `(repo_id, snapshot_id)` before rendering — `internal/cli` and `internal/snapshot` — unknown IDs stop with not-found before any current-cache rebuild or current-index query
- `diff-snapshot` resolves stored base and head snapshots before comparison — `internal/snapshot` — both IDs must resolve and pass stored-row consistency checks before deterministic diff output is produced
- Stale-index detection occurs only at snapshot creation — `internal/cache`, `internal/index`, `internal/snapshot` — stored warnings compare chunk source/index metadata against source/sync state and are carried forward unchanged into export/diff
- Consistency checks run from stored rows only — `internal/snapshot` via `internal/cache` — export/diff count `snapshot_chunks` rows and recompute canonical hashes for the resolved snapshot IDs without scanning records or rebuilding chunks

## Component Interactions
- `internal/index` -> `internal/snapshot` — provides deterministic repo-scoped chunk rows with source ranges, source/content hashes, index build metadata, heading paths, ordinals, and chunk text for snapshot creation
- `internal/cache` -> `internal/snapshot` — owns SQLite transactions and typed query methods for `snapshots` and `snapshot_chunks`, including WAL-safe storage, stored-row consistency reads, and typed not-found/consistency errors
- `internal/snapshot` -> `internal/cli` — exposes `SnapshotStore` product methods returning `SnapshotExport`, `DiffSnapshotResult`, warnings, and typed errors; CLI renders results without rebuilding snapshots itself
- `internal/cache` -> `internal/snapshot` — provides repo-scoped source, identity, link, and sync summaries only during `CreateSnapshot`; export/diff consume only persisted snapshot rows after ID resolution
