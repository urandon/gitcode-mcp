# Component Design: GitCode Adapter

## Summary
GitCode Adapter is detailed because the approved architecture assigns this component the live/fixture/test provider seam, credential-gated live validation, typed provider failures, pagination behavior, redaction, and write-confirmation contract. Current `internal/gitcode` already contains a live `Client`/`HTTPClient`, endpoint builders, pagination helpers, domain models, typed errors, and write methods, but it does not yet provide the required provider construction boundary, fixture provider, live-admission invariant, redacted capture surface, or confirmed write metadata.

## Top-Level Alignment
This component is the only boundary allowed to talk to GitCode during explicit sync, write, or live-validation operations. It supplies canonical issue/wiki/comment/search data to cache-sync owners, trusted write confirmations to CLI-write/audit owners, and offline fixture behavior for credential-free validation.

### Task 1: Provider seam add
Outcome IDs: outcome-13
Outcome Role: primary_product
Decommission IDs: none
Change Type: add
Description: Add an `internal/gitcode` provider seam over the existing live client so live and fixture implementations expose the same issue, wiki, comment, search, repo, auth, pagination, error, and redaction behavior. The seam keeps all live network access explicit and gives tests a credential-free fixture path. Attachments and label methods already exposed by `Client` are left as existing passthrough/non-blocking behavior and are not part of the new fixture or write-confirmation minimum for this iteration.
Existing Behavior / Reuse: Reuse `internal/gitcode.Client`, `HTTPClient`, `Config`, endpoint builders, request/response structs in `models.go`, `Page[T]`, `PageState`, `PaginationStrategy`, `getPaged`, existing write methods, and typed errors in `errors.go` where they fit. Confirmed absent from the current component are `Provider`, `ProviderConfig`, `ProviderMode`, `LiveAllowed`, `NewLiveProvider`, `NewFixtureProvider`, `NewUnavailableProvider`, fixture-backed provider methods, repo/auth probe models, redacted capture models, adapter-local live admission before `NewHTTPClient`, pagination loop/malformed guards, and a durable live-test gate using `GITCODE_LIVE_TEST`.
Detailed Design: Add a component-local `Provider` interface in `internal/gitcode` with non-generic domain methods: `ProbeAuth(context.Context, AuthProbeRequest) (AuthProbeResult, error)`, `GetRepo(context.Context, RepoRequest) (Repo, error)`, `ListIssues`, `GetIssue`, `ListIssueComments`, `ListWikiPages`, `GetWikiPage`, `Search`, `CreateIssue`, `UpdateIssue`, `CreateIssueComment`, `CreateWikiPage`, and `UpdateWikiPage`; keep attachment/label methods outside the required provider seam unless reused by an existing `Client` adapter without widening test obligations. Add `ProviderMode`, `ProviderConfig`, `FixtureConfig`, `RepoRequest`, `Repo`, `AuthProbeRequest`, `AuthProbeResult`, `ErrProviderUnavailable`, `ErrValidationFailed`, `ErrPaginationMalformed`, `ErrPaginationLoop`, and `RedactedCapture` as `internal/gitcode` concepts. Implement `NewLiveProvider(ProviderConfig)` with the adapter-local admission invariant `cfg.Mode == ProviderModeLive && cfg.LiveAllowed && strings.TrimSpace(cfg.Token) != ""`; token-present-but-not-live-allowed and live-allowed-without-token both return `ErrProviderUnavailable` before calling `NewHTTPClient`. Implement `NewFixtureProvider(FixtureConfig)` that loads sanitized JSON into the same domain structs and returns deterministic auth probe, repo, issue page/list/get, comment list, wiki page/list/get, search, conflict, rate-limit, malformed-page, and pagination fixtures without token lookup or HTTP construction. Keep `getPaged` as a package-level generic helper rather than a generic interface method; live list/search methods continue to call it, and fixture list/search methods reuse the same pagination algorithm through either a shared `paginateFixture[T]` helper or an injected page-fetch function that accepts `PageState` and `PaginationStrategy`. Change the helper invariants so page-number and cursor strategies terminate only on explicit no-next state, reject duplicate or non-advancing states, reject negative page/per-page metadata, surface rate limits as `ErrRateLimited` with `RetryAfter` and `RawRetryAfter`, and return zero accumulated records on later-page failure unless a caller opts into a typed partial-result API. Add redaction functions for headers, URLs, JSON bodies, error payloads, cookies, authorization values, bearer tokens, owner/repo names, and unapproved hostnames; live capture writers accept only `RedactedCapture`, never raw response bytes. Add `RequireLiveProviderForTest` or equivalent test support that skips unless `GITCODE_LIVE_TEST=1` and a token is present, with any existing `GITCODE_TEST_TOKEN` treated only as a transitional alias if already used.
Acceptance Criteria: A maintainer runs adapter contract tests with no credentials against `NewFixtureProvider`; target API surface `internal/gitcode.Provider` returns deterministic repo/auth, issue list/get, comment list, wiki list/get, search, pagination, auth-error, conflict, and rate-limit responses with no external network, evidenced by `go test ./internal/gitcode ./internal/testnet`. A maintainer runs live-provider admission tests; `NewLiveProvider` with token-present-but-live-not-allowed and live-allowed-without-token returns `ErrProviderUnavailable` before `NewHTTPClient`, evidenced by local unit tests using an HTTP-construction sentinel. A maintainer enables `GITCODE_LIVE_TEST=1` with a token against a test provider; live validation either skips safely or writes only `RedactedCapture` artifacts that pass fixture sanitization, evidenced by credential-gated adapter tests.

### Task 2: Write confirmation add
Outcome IDs: outcome-9
Outcome Role: primary_product
Decommission IDs: decommission-2
Change Type: add
Description: Strengthen `internal/gitcode` writes so issue, comment, and wiki operations cannot report fake success. Dry-run remains owned by CLI write, but every adapter live write returns either a confirmed provider result or a typed error. Mock success behavior is test support only and cannot be selected by product CLI/config provider selection.
Existing Behavior / Reuse: Reuse existing `CreateIssue`, `UpdateIssue`, `CreateIssueComment`, `CreateWikiPage`, `UpdateWikiPage`, `WriteOptions`, `WriteResult[T]`, `GenerateIdempotencyKey`, `writeJSON`, endpoint builders, and typed errors such as `ErrConflict`, `ErrAuthExpired`, `ErrForbidden`, `ErrRateLimited`, `ErrNetworkUnavailable`, `ErrPayloadTooLarge`, and `ErrNotFound`. Current `WriteResult[T]` only carries `Record` and `IdempotencyKey`; it lacks `Confirmed`, provider status, response hash, confirmed timestamp, operation, target, and operation-specific identity minima. Current code also has no unavailable provider and no adapter-level negative invariant proving disabled, local queued, unavailable, or mock-selected product paths cannot return success-shaped write data.
Detailed Design: Extend or wrap `WriteResult[T]` with confirmation metadata: `Confirmed bool`, `Operation`, `Target`, `ProviderStatus`, `RemoteID`, `RemoteNumber`, `RemoteSlug`, `RemoteRevision`, `ParentIssueNumber`, `ParentIssueID`, `IdempotencyKey`, `ResponseHash`, `ConfirmedAt`, and optional redacted `ProviderPayloadFingerprint`. Add request validation functions used before network calls: owner/repo required for all writes, title required for issue create, positive issue number required for issue update/comment, comment body required, wiki title/body plus slug-or-returned-slug required for create, wiki slug required for update, and idempotency key either supplied or generated. Change `writeJSON` or add `writeConfirmedJSON` so it computes a SHA-256 response hash, classifies every non-2xx response into typed adapter errors, redacts remote conflict payloads before attaching them to `ErrConflict`, decodes 2xx bodies, and constructs `Confirmed=true` only after operation-specific minima pass. Confirmation minima are: issue create requires nonempty issue `id` and positive `number`; issue update requires nonempty issue `id` and returned `number` matching the request; comment create requires nonempty comment `id` plus parent issue identity from response `IssueID` or request `Number`; wiki create requires returned slug/path plus either wiki `id` or `version`; wiki update requires returned slug/path matching the request plus wiki `id` or `version`; every confirmed result requires nonempty idempotency key, nonempty response hash, provider status, operation, target, and confirmed timestamp. Add `NewUnavailableProvider(reason)` whose write methods always return `ErrProviderUnavailable` or `ErrNetworkUnavailable` and never return decoded records, nil error, `Confirmed=true`, audit-ready metadata, or success-shaped data. Move success-producing mock provider support under adapter test support such as `internal/gitcode/gittest` or build/test-only constructors; product provider selection accepts only live, fixture, or unavailable modes and has a negative invariant that no CLI/config mode can select a success mock. Keep attachment and label writes as existing passthrough/non-blocking if present; they are not included in the required confirmed write surface for this iteration.
Acceptance Criteria: A developer invokes `CreateIssue`, `UpdateIssue`, `CreateIssueComment`, `CreateWikiPage`, and `UpdateWikiPage` through the adapter API against a stubbed external provider; each valid 2xx response returns `Confirmed=true` with operation, target, required remote identity, idempotency key, response hash, provider status, and confirmed timestamp, evidenced by adapter API tests. A developer invokes the same API for conflict, auth, forbidden, rate-limit, network, malformed success response, validation failure, and unavailable-provider scenarios; each returns the matching typed error with no confirmed result, no decoded success-shaped record, and no audit-ready metadata, evidenced by stubbed-external-provider tests. A CLI/write owner requests provider selection through product runtime config; success mocks are not selectable while unavailable providers cannot be audited as success, evidenced by CLI/API tests with external-provider mocks and provider-selection tests.
Workload: 1.1 MM

## Cross-Cutting Constraints
- Live network access is explicit and credential-gated — cache-first reads and routine tests must not instantiate live HTTP, while explicit sync/write/live-validation call sites may do so only with live admission and token
- Durable artifacts are redacted before persistence — fixture and live-capture evidence must not contain credentials, cookies, private hostnames, owner/repo names, raw tokens, or unsanitized provider bodies
- Provider success is confirmation-based — write callers may audit success only after a confirmed adapter response with operation-specific identity minima
- Pagination is all-or-typed-error — malformed, looping, duplicate, rate-limited, or later-page failures must not look like successful partial pages unless explicitly typed as partial/error results
- Mock success is test-only — no product CLI/config provider selection can choose a success mock or convert unavailable provider behavior into successful write state

## Data And Control Flow
- Fixture validation — `NewFixtureProvider` loads sanitized fixtures into provider methods, adapter contract tests call repo/auth/issue/comment/wiki/search/page operations, and no token lookup or HTTP client construction occurs
- Product live sync/write — explicit caller passes `ProviderConfig{Mode: ProviderModeLive, LiveAllowed: true, Token: ...}`, `NewLiveProvider` admits and constructs `HTTPClient`, operation methods decode domain structs or typed errors, and redaction runs before any capture writer receives response data
- Write execution — request validation runs before network, confirmed write transport performs idempotent HTTP write, provider response is decoded and checked against confirmation minima, then either confirmed result or typed error is returned
- Pagination execution — list/search methods call a package-level helper using `PageState` and `PaginationStrategy`, the helper advances page/cursor state, detects loops and malformed continuation, and returns zero successful records on untyped page failure

## Component Interactions
- `gitcode-adapter` -> `cache_sync` — sync consumes repo/auth, issue, comment, wiki, and search read methods returning canonical domain structs or typed errors for idempotent cache population
- `gitcode-adapter` -> `cli_write` — live write commands consume confirmed write results or typed errors; dry-run remains outside the adapter and unavailable providers cannot be reported as remote success
- `gitcode-adapter` -> `docs_dogfood` — fixture/live validation and redaction behavior provide public-safe evidence for fixture capture and optional live probing docs without making docs owned by this component
- `gitcode-adapter` -> `testnet` — adapter tests use test network guards and live-test skip helpers so offline validation cannot leak to external GitCode hosts

## Rationale
The component is affected because `internal/gitcode` is the existing GitCode API boundary and already owns live HTTP operations, endpoint builders, pagination, typed errors, and write calls. The approved architecture requires this boundary to become the reliable provider seam for live and fixture implementations and the sole producer of trusted write confirmations and typed provider failures.
