# Component Design: Knowledge Indexer

## Summary
The knowledge-indexer component adds the missing `internal/index` package for deterministic cache-derived indexing, markdown parsing, citation/snippet anchors, stale-link detection, and chunk generation. It owns component-local parsing and build logic that replaces hand-maintained markdown indexes with reproducible derived records.

## Top-Level Alignment
This component implements the approved `internal/index` responsibility: full and incremental derived-index builds, content-hash comparison, frontmatter/heading/link parsing, stale detection, and deterministic chunks. It remains a peer of `internal/cache`; `internal/service` invokes it and supplies cache-backed interface implementations.

## Tasks

### Task 1: Add index builder entities
Outcome IDs: outcome-9, outcome-11
Outcome Role: primary_product
Decommission IDs: decommission-2
Change Type: add
Description: Add the component-local `internal/index` package with builder, parser, chunker, citation-anchor, derived-projection, and staleness entities required by the approved architecture. The package converts cached markdown-like source records into derived indexes and deterministic chunks without owning SQLite, CLI dispatch, MCP transport, or remote GitCode behavior. This replaces hand-maintained markdown index files as the product runtime source for task backlogs, source ledgers, track views, backlink graphs, broken-link reports, stale-index reports, and snippet/citation anchors.
Existing Behavior / Reuse: Existing source confirms `internal/index` is absent; only `internal/cli` exists, so there is no index builder, chunker, stale-checker, citation-anchor emitter, or derived projection implementation to reuse. Reuse the repository’s documented cache-first model and current Go standard-library style. Consume cache data through narrow component-local interfaces supplied by service rather than importing peer package internals or duplicating database access. Hand-maintained plaintext indexes are replaced for product runtime by generated derived projections; fixture markdown sources remain valid test input, but runtime derived commands must read generated cache/index data rather than treating those plaintext indexes as authoritative.
Detailed Design: Add an `index` package centered on `Builder`, `SourceReader`, `DerivedWriter`, and `DerivedLinkReader` contracts. `SourceReader` supplies normalized `SourceRecord` values with stable id, kind, path, body, metadata, updated timestamp, aliases, remote aliases, and previous indexed content hash. `DerivedWriter` accepts `DerivedLink`, `BacklinkRow`, `BrokenLink`, `CollisionDiagnostic`, `CitationAnchor`, `Chunk`, `SourceLedgerRow`, `TaskBacklogRow`, `TrackRow`, `AcceptanceRow`, `OpenQuestionRow`, and per-source `IndexState`. `DerivedLinkReader` supplies current derived links for `StaleCheck` when service wants a read-only stale scan. `internal/index` must not import `internal/cache`; `internal/service` owns cache-backed adapters that implement these interfaces.

Add `FullBuild(ctx, reader, writer)`, `IncrementalBuild(ctx, reader, writer)`, and `StaleCheck(ctx, reader, links)` functions. `FullBuild` processes every source. `IncrementalBuild` processes only records whose body-derived content hash differs from stored index state and returns a build report with processed count, skipped count, rewritten derived-row count, collision count, and broken-link count. Unchanged sources must not rewrite derived rows, chunks, anchors, or index state.

Implement a markdown parser pipeline with component-local data types `ParsedSource`, `Frontmatter`, `Heading`, `Link`, `ResolvedLink`, `Status`, `StableID`, `Alias`, `DerivedRecord`, `BrokenLink`, `CollisionDiagnostic`, `StaleReport`, `CitationAnchor`, and `Chunk`. The parser extracts optional frontmatter key/value metadata, heading hierarchy, stable ids and aliases, task/status markers, outbound markdown links, wiki-style ids, open-question markers, and acceptance-related sections. Link resolution uses aliases supplied by `SourceReader`; resolved aliases produce `DerivedLink` rows and reversed `BacklinkRow` rows, while unresolved links produce `BrokenLink` rows.

Add the component-local citation/snippet anchor model as `CitationAnchor`. Each anchor maps `anchor_id`, `source_id`, `content_hash`, `byte_start`, `byte_end`, `line_start`, `line_end`, `heading_path`, `anchor_kind`, `title`, and optional `derived_row_id`. Anchors are emitted during parsing for headings, task/status blocks, acceptance sections, open-question blocks, link targets, and chunk boundaries. They are handed to `DerivedWriter` alongside chunks and derived rows; later `service.GetSnippet` and citation flows consume persisted anchors through cache-backed readers to choose the nearest heading/context range, then retrieve authoritative source text by `source_id + content_hash + byte offsets`. Ownership stays in `internal/index` for anchor production and in `internal/service`/cache-backed adapters for storage and lookup.

Implement deterministic chunking as `ChunkSource(source SourceRecord, parsed ParsedSource) []Chunk`. Chunk ids are SHA-256 over `source_id`, `content_hash`, and `byte_start`. Chunks store source id, content hash, byte start/end, line start/end, heading path, raw text, normalized text, inherited metadata, outbound links, resolved aliases, and matching citation anchor id when the chunk boundary creates or reuses an anchor. The chunker computes byte offsets against the original UTF-8 source body, line ranges from normalized newline boundaries, and heading paths from the active heading stack; re-running on identical input must produce byte-identical chunk ids and payloads.

Parser and resolver invariants are part of the build contract. Malformed frontmatter does not stop parsing the body; it emits a `CollisionDiagnostic` or parse diagnostic with source id and line range, and metadata falls back to empty/partial parsed values. Duplicate stable ids in different sources are build-blocking collision diagnostics for affected ids; the writer receives diagnostics and no alias/index-state commit occurs for colliding sources. Duplicate aliases for the same source are deduplicated; duplicate aliases across different sources produce ambiguous-alias diagnostics and links using that alias become `BrokenLink` rows with reason `ambiguous_alias`. Ambiguous aliases never resolve silently. Unresolved relative paths produce `BrokenLink` rows with raw link text, source path, line range, and reason `unresolved_relative_path`. CRLF and CR newlines are normalized for line counting and normalized text, while byte offsets always refer to the original body bytes. Invalid UTF-8 is replaced only in normalized text; byte offsets remain original byte indexes and diagnostics identify the invalid range.

Implement stale detection as a pure index scan over current source ids, aliases, and derived links. `StaleCheck` returns a structured report with total stale backlink count, affected source ids, unresolved targets, broken raw link text, ambiguous aliases, and stale anchor references, and must not mutate records. The negative decommission invariant is enforced by making derived outputs for source ledger, track index, task backlog, acceptance ledger, open questions, backlink graph, broken-link report, and citation anchors originate only from cached source records plus parser output; no product index build path reads or trusts hand-maintained markdown index files as already-derived state.
Acceptance Criteria: A developer triggers the product index path by running `go test ./internal/index/... -run TestIndexPipeline` against cache-backed fixture sources; `FullBuild` returns success, derived projections contain source-ledger/task-backlog/backlink/broken-link data, `StaleCheck` returns a JSON-serializable stale backlink count and affected source ids, and `IncrementalBuild` on unchanged sources reports no rewrites and zero new stale items. A developer triggers the chunking product path by running `go test ./internal/index/... -run TestChunkDeterminism`; a markdown source produces chunks with correct byte offsets, line ranges, heading paths, inherited metadata, outbound links, resolved aliases, and a second run over the same source produces identical chunk ids. A developer runs `go test ./internal/index/... -run TestCitationAnchors`; heading/task/acceptance anchors include correct source id, content hash, byte offsets, line ranges, heading paths, and are emitted through `DerivedWriter` so service/cache adapters can satisfy `GetSnippet` and citation lookup by anchor. A developer runs `go test ./internal/index/... -run TestParserLinkEdgeCases`; malformed frontmatter yields parse diagnostics and body parsing continues, duplicate ids produce collision diagnostics and block affected commits, duplicate same-source aliases dedupe, ambiguous cross-source aliases produce ambiguous diagnostics and broken-link rows, unresolved relative paths produce broken-link rows, CRLF line ranges are normalized, and UTF-8 byte offsets remain original-byte accurate. A developer runs `go test ./... -short` and the index tests complete without network access, proving cache-first behavior used by `gitcode-mcp index --full`, `gitcode-mcp index --incremental`, and `gitcode-mcp stale-index`.

## Cross-Cutting Constraints
- Cache-first execution — index builds, citation-anchor production, chunking, and stale checks operate from local cache records and never require live GitCode access.
- Peer-package separation — `internal/index` must not import `internal/cache`; service coordinates cache-backed `SourceReader`, `DerivedWriter`, and `DerivedLinkReader` adapters so `cache`, `gitcode`, and `index` remain data-layer peers.
- Deterministic provenance — chunks, anchors, and derived rows must be reproducible from source id, content hash, offsets, metadata, and links so exports and future RAG composition can cite authoritative cache records.
- Runtime decommission replacement — generated derived projections replace hand-maintained markdown indexes for product behavior, while raw markdown sources remain ingest inputs.

## Data And Control Flow
- Service starts `FullBuild` or `IncrementalBuild` — `internal/service` and `internal/index` — service supplies cache-backed `SourceReader` and `DerivedWriter`; index reads normalized sources, computes hashes, parses content, emits anchors/chunks/projections, and writes derived rows through the writer.
- Parser normalizes source content — `Builder` and parser entities — frontmatter, headings, statuses, ids, aliases, outbound links, open questions, acceptance sections, and citation anchors are extracted before projection generation.
- Link resolver emits derived links — parser and resolver entities — resolved links become outbound/backlink rows, ambiguous or unresolved links become diagnostics and broken-link rows, and duplicate ids block affected source commits.
- Chunker emits deterministic chunks — `ChunkSource`, `CitationAnchor`, and `DerivedWriter` — chunk ids and anchor ids are computed after content hash and byte offsets are known, then stored with heading path and inherited metadata.
- Stale checker reads derived links — `StaleCheck` and service-supplied `DerivedLinkReader` — missing targets, unresolved aliases, stale anchor references, and broken raw links produce stale report entries without mutating cache state.
- Citation/snippet lookup consumes anchors — `internal/service` and cache-backed readers — service resolves an id or anchor, reads the persisted `CitationAnchor`, then calls cache snippet retrieval by source id/content hash/byte range while index remains only the anchor producer.

## Component Interactions
- `internal/service` -> `internal/index` — service invokes `FullBuild`, `IncrementalBuild`, `StaleCheck`, and chunk/anchor production, supplies context cancellation, and passes cache-backed reader/writer adapters.
- `internal/service` -> `internal/index.SourceReader` — service implements the reader using cache data and alias maps; `internal/index` sees only normalized source records and never imports `internal/cache`.
- `internal/index.DerivedWriter` -> `internal/service` — index emits links, chunks, citation anchors, derived projections, diagnostics, and index-state updates through the writer; service persists them through cache-owned APIs.
- `internal/index.DerivedLinkReader` -> `internal/service` — stale checks consume service-supplied derived-link snapshots without mutating cache and without package-level dependency on cache schema.
- `internal/cli` -> `internal/service` — CLI commands such as `index --full`, `index --incremental`, `stale-index`, `tasks`, `tracks`, and `link-check` consume index results only via service methods.
- future RAG layer -> `internal/index` data — semantic retrieval may use stored chunks and anchors as candidates, while `resolve_id`, `get_source`, `get_snippet`, backlinks, and sync status remain authoritative citation paths.

## Rationale
The approved architecture materially affects this component because `internal/index` is currently missing and the Component Impact delta requires detailed implementation of builder, chunker, derived projection, stale-check, and citation/snippet behavior. The work is component-local, traceable only to request Tasks 9 and 11, and supplies generated data needed to decommission hand-maintained plaintext indexes.
