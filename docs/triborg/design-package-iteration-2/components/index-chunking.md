# Component Design: Index & Chunking

## Summary
Index & Chunking is materially affected because the approved architecture requires deterministic, repo-scoped chunk generation and explicit stale/missing-index reporting. Existing `internal/index` concepts are reusable, but the component lacks explicit policy selection, shared chunk query contracts, sliding-window chunking, repo/snapshot chunk metadata, and version-aware index freshness.

## Top-Level Alignment
This component owns the cache-to-index transformation used by CLI reads, MCP tools, doctor diagnostics, and snapshot export. It must emit deterministic chunks with source ranges and hashes, expose a single query result model consumed by CLI and MCP, and classify stale/missing index state instead of allowing downstream surfaces to silently omit chunks or citations.

## Tasks

### Task 1: ChunkPolicy and Query API
Outcome IDs: outcome-12
Outcome Role: primary_product
Decommission IDs: none
Change Type: add
Description: Add a policy-aware chunking engine and component-owned chunk query API inside Index & Chunking. The component already derives chunks from parsed source records, but it must explicitly support default heading-delimited chunking, max-section fallback, optional sliding-window chunking, and shared list/search/snippet result shapes for CLI and MCP. The chunking engine owns deterministic chunk identity, source ranges, content hashes, policy metadata, and warning metadata handed to downstream read surfaces.
Existing Behavior / Reuse: Reuse existing `Builder`, `SourceRecord`, `ParsedSource`, `Chunk`, `ChunkSource`, `FullBuild`, `IncrementalBuild`, heading parsing, `ContentHash`, citation anchors, normalization, outbound-link extraction, and deterministic chunk ID concepts. Confirmed absent or incomplete in `internal/index`: explicit `ChunkPolicy`, `ChunkOptions`, sliding-window chunking, configurable max-size behavior, repo_id/snapshot_id/policy on chunk records, and index-owned query request/result contracts for `list_chunks`, `search_chunks`, and `get_snippet`.
Detailed Design: Add component-local `ChunkPolicy` values `heading` and `sliding_window`, plus `ChunkOptions` with `Policy`, `MaxBytes`, `WindowBytes`, `OverlapBytes`, and stable defaults; the zero value must preserve current `ChunkSource(source, parsed)` heading behavior with the existing soft limit semantics. Change `Builder` from an empty struct to carry `ChunkOptions`, add `NewBuilder(options ChunkOptions) Builder`, `FullBuildWithOptions(ctx, reader, writer, options)`, and `IncrementalBuildWithOptions(ctx, reader, writer, options)`, while keeping existing `FullBuild`, `IncrementalBuild`, and `ChunkSource(source, parsed)` as compatibility wrappers that call default options. Add `ChunkSourceWithOptions(source, parsed, options)` and route `deriveSource` through the builder’s options so CLI/index callers can pass one non-default path such as `ChunkOptions{Policy: SlidingWindow, WindowBytes: n, OverlapBytes: m}` without duplicating chunking logic in CLI or MCP.

Extend index-owned `SourceRecord` and `Chunk` data classes with optional `RepoID`, `RecordID`, `SnapshotID`, and `Policy`; every emitted chunk must copy upstream repo/snapshot context unchanged when present and must contain non-empty source id, content hash, byte range, line range, deterministic ID, normalized text, heading path, citation anchor id, policy, and inherited metadata. Implement heading policy as current heading split plus max-section fallback at paragraph boundaries first, then line boundaries, and never split inside fenced code unless a single fenced block exceeds the hard limit. Implement sliding-window policy over line-aligned windows using deterministic byte starts and configured overlap, preserving citation ranges and heading context from the first byte in each window. Chunk IDs must include source id, content hash, policy, byte start, max/window/overlap parameters, and snapshot id when present so repeated indexing of identical input with identical options produces byte-identical chunks and different policies cannot collide.

Add a component-owned chunk query contract with data classes such as `ChunkQuery`, `SnippetQuery`, `ChunkSearchQuery`, `ChunkQueryResult`, `ChunkResult`, and `IndexWarning`. `ChunkQuery` must support `RepoID`, `SourceID`, `RecordID`, `SnapshotID`, `Policy`, `Limit`, and `Offset`; `ChunkSearchQuery` adds normalized query text; `SnippetQuery` supports `RepoID`, `SourceID` or `RecordID`, `ChunkID`, byte offset/length, or line start/end. Add an interface such as `ChunkIndexReader` with `ListChunks(ctx, ChunkQuery)`, `SearchChunks(ctx, ChunkSearchQuery)`, and `GetSnippet(ctx, SnippetQuery)` returning `ChunkQueryResult`. Results must include deterministic ordering by repo_id, source_id, policy, byte_start, chunk_id; source ranges; content hash; heading path; policy; normalized text or snippet text; pagination metadata; and warning metadata from the freshness classifier. CLI `list-chunks`/`get-snippet` and MCP `list_chunks`/`search_chunks`/`get_snippet` must consume this shared result shape rather than maintaining separate filtering, ordering, or warning logic.
Acceptance Criteria: A developer indexes sanitized fixture issue, wiki, and changelog records through the index runtime using default options and then using `sliding_window`; CLI `list-chunks`/`get-snippet` and MCP `list_chunks`/`search_chunks`/`get_snippet` return chunks through the shared `ChunkQueryResult` shape. Visible responses include deterministic IDs, repo/source/record/snapshot metadata when present, policy, content hashes, byte and line ranges, heading path, normalized text or snippet text, pagination fields, and warning metadata. Executable evidence is local indexing tests plus CLI and MCP read tests over sanitized fixtures proving two repeated runs produce identical chunk IDs/ranges/content, both policies remain queryable, limit/offset ordering is stable, and CLI/MCP list/snippet responses are parity-compatible over the same query result model.

### Task 2: IndexFreshness warnings
Outcome IDs: outcome-1, outcome-5, outcome-6, outcome-11
Outcome Role: primary_product
Decommission IDs: decommission-8
Change Type: add
Description: Add version-aware content-index freshness detection owned by Index & Chunking. Existing stale checking focuses on derived links and broken backlinks, but doctor, CLI, MCP, and snapshot surfaces need explicit missing/stale-index warnings for records whose content hash, remote version, sync version, or updated timestamp is newer than indexed chunks. This replaces product-visible silent omission of chunks/citations with deterministic warning metadata.
Existing Behavior / Reuse: Reuse existing `StaleCheck`, `StaleReport`, `IndexState`, `SourceRecord.PreviousIndexedHash`, chunk content hashes, citation anchors, and derived-link stale diagnostics. Replace the product-visible behavior where missing chunks or stale citation ranges can be interpreted as empty successful results; keep raw empty slices only for internal helpers that are wrapped by freshness-aware public query methods.
Detailed Design: Add index-owned freshness data classes such as `IndexFreshnessReport`, `IndexFreshnessRecord`, and `IndexWarning` with fields for repo id, source id, record id, snapshot id, current content hash, indexed content hash, current remote version, indexed remote version, current sync event id or version token, indexed sync event id or version token, current updated timestamp, indexed updated timestamp, chunk count, citation count, state, and warning code. Extend `SourceRecord` or add a companion `SourceRevisionState` supplied by `cache_sync` with `RepoID`, `RecordID`, `RemoteRevision`, `SyncRevision`, `SyncEventID`, and `UpdatedAt`; extend `IndexState` with `RepoID`, `RecordID`, `RemoteRevision`, `SyncRevision`, `SyncEventID`, `SourceUpdatedAt`, `Policy`, and `IndexedAt`. Add an `IndexStateReader` interface that supplies indexed state, chunks, and citation anchors for a repo/source set, then extend or complement `StaleCheck` with `FreshnessCheck(ctx, sourceReader, indexStateReader, linkReader)` so link staleness and content-index staleness are classified together without live network access.

Freshness comparison precedence must be deterministic: if a record has no indexed state or no chunks for the requested policy, classify as `missing_index`; else if current content hash differs from indexed content hash, classify as `stale_by_content`; else if remote version differs, sync version differs, or current updated timestamp is after indexed source updated timestamp, classify as `stale_by_revision`; else if content and version are fresh but derived links, broken backlinks, or citation anchors are stale, classify as `link_stale_only`; otherwise classify as `fresh`. Public warning codes should include `missing_index`, `stale_index`, `stale_index_revision`, and `link_stale_only`; `missing_index` outranks all stale classifications, content stale outranks version stale, and link-only stale never masks fresh chunk/citation state. Integrate the classifier into the query contracts from Task 1 so `ListChunks`, `SearchChunks`, and `GetSnippet` return warning metadata alongside empty or stale chunk/citation results.

For `decommission-8`, disable unqualified product success for absent chunks or citations: CLI, MCP, doctor, and snapshot export paths must call freshness-aware query/report methods and surface the warning code when chunks or citations are missing/stale. Internal low-level readers may still return raw empty slices only when not exposed directly to product runtime. Deterministic record ordering for freshness reports must be by repo_id, source_id, record_id, state precedence, and warning code so CLI/MCP/snapshot consumers present stable output.
Acceptance Criteria: A developer prepares sanitized fixture cache states for records with no chunks, records changed after indexing by content hash, records changed only by remote/sync version or updated timestamp, records with fresh chunks, and records with fresh chunks but stale/broken links. When the developer runs CLI `stale-index`, `cache-status`, `get-snippet`, MCP `stale_index_report`, MCP `get_snippet`, and CLI `export-snapshot`, responses visibly include `missing_index` warnings for unindexed records, `stale_index` or `stale_index_revision` warnings for changed records, `link_stale_only` diagnostics for link-only stale records, and no freshness warning for fresh records. `get-snippet` and `export-snapshot` must not silently return empty chunks or omit citations without warning metadata. Executable evidence is index runtime tests, CLI integration tests, MCP JSON-RPC tests, and snapshot export tests over fixture cache states proving fresh, missing, stale-by-content, stale-by-version, and link-stale-only classifications produce expected visible fields and deterministic ordering.
Workload: 1.2 MM

## Cross-Cutting Constraints
- Chunk output must be deterministic for identical source content, policy, and options because CLI, MCP, and snapshot parity depend on stable chunk IDs, ranges, and ordering
- Routine index reads must be cache-first and must not require live network access; this component consumes cached source records, indexed state, chunks, links, and citation anchors only
- Missing or stale index state must be surfaced as explicit warnings so snapshot export and read surfaces cannot silently omit chunks or citations
- CLI and MCP chunk/snippet parity must be enforced through a shared index query result model, not duplicated presentation-specific query logic
- RAG embeddings and retrieval ranking remain deferred; chunk records may be RAG-ready but this component must not add vector retrieval behavior

## Data And Control Flow
- Cache source records enter the index builder — `Builder` parses source bodies, computes content hashes, selects `ChunkPolicy`, and emits derived chunks/citation anchors — ordering is deterministic by repo_id, source_id, policy, and chunk byte range
- Chunk policy execution emits chunk metadata — `ChunkSourceWithOptions` copies repo/record/snapshot metadata from source context, computes byte/line ranges, normalized text, outbound links, policy, and deterministic IDs — no live adapter is involved
- CLI and MCP list/search/snippet calls consume the same `ChunkQueryResult` model — `cli_read` and `mcp_server` may format the result differently, but filtering, deterministic ordering, source ranges, content hashes, policy matching, pagination, and freshness warning classification are owned by `index_chunking`
- Freshness checking compares current source hashes and cache-supplied version state with indexed state — `FreshnessCheck` classifies fresh, missing, stale-by-content, stale-by-version, or link-stale-only records and returns warning records for downstream consumers
- Snapshot export and MCP/CLI reads request chunks or freshness reports — index returns chunks plus warning metadata — consumers decide presentation but not warning semantics or state precedence

## Component Interactions
- `cache_sync` -> `index_chunking` — provides repo-scoped source records, record ids, remote revisions, sync version/event state, updated timestamps, previous indexed hash/state, and persists derived chunks/citation anchors emitted by the builder
- `index_chunking` -> `cli_read` — exposes `ChunkQueryResult` and `IndexFreshnessReport` for snippets, stale-index, cache-status, list-chunks, search-chunks, and read outputs; CLI must not reimplement chunk filtering or warning classification
- `index_chunking` -> `mcp_server` — exposes the same `ChunkQueryResult` and stale-index report semantics used by CLI so MCP responses remain parity-compatible for list/search/snippet surfaces
- `index_chunking` -> `snapshot_diff` — provides chunks, citation ranges, policy metadata, and missing/stale-index warnings for export-snapshot and diff-related validation
- `config_credential` -> `index_chunking` — doctor/runtime audit consumes index freshness state and warning counts; index owns classification, doctor owns display

## Rationale
The configured component impact marks Index & Chunking as detailed and names two concrete deltas: deterministic chunk policy/index population and explicit stale/missing-index reporting. Existing `internal/index` code provides reusable parsing, chunking, builder, and stale-link anchors, but the approved architecture requires additional component-local APIs and invariants for policy selection, query parity, and version-aware freshness.
