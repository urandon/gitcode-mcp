# Component Design: Repository Binding

## Summary
Repository binding is affected because the approved architecture makes `repo_id` the master scope key for configured repositories, aliases, cache access, MCP filters, sync/write routing, and snapshot selection. The current implementation has source, identity, CLI, MCP, and cache concepts, but repository scope is absent, so this component needs concrete product work.

## Top-Level Alignment
The component owns the `repo_id` boundary required by Task 2 and decommission-resolution-6. It supplies repository metadata and scoped identity resolution to cache, CLI, MCP, sync, write, and snapshot consumers without performing live GitCode calls itself.

## Tasks

### Task 1: Repo Registry Add
Outcome IDs: outcome-2
Outcome Role: primary_product
Decommission IDs: decommission-6
Change Type: add
Description: Add the repository registry as the component-local source of truth for configured GitCode repositories. The registry owns `repo_id`, owner/name, API base URL, enabled scopes, display name, repository aliases, and repo status diagnostics. It replaces the current implicit single-cache identity model where cache sources and aliases have no repository boundary.
Existing Behavior / Reuse: Current cache source and identity models expose `Source`, `Identity`, `RemoteAlias`, and `Store.ResolveAlias`, but no `repo_id` field, no `repos` registry, and no repository-alias concept. Current cache schema stores `sources(id PRIMARY KEY)` and `identity_map PRIMARY KEY(alias_type, alias)`, making aliases globally unique. Current CLI command handling has read commands and a `--repo` option used as repository name context for write-like commands, but no `repo add`, no `repo status`, and no configured repo binding product surface. Reuse existing cache error classes, CLI formatting patterns, and typed conflict/not-found behavior; leave config loading, adapter calls, and cache record storage implementation owned by their components except for the repo-binding schema contract.
Detailed Design: Add a component-local repository binding package concept with `RepositoryBinding`, `RepositoryScope`, `RepositoryAlias`, `RepositoryRegistry`, `RepositoryStatus`, and `RepoBindingError`. `RepositoryBinding` contains `RepoID`, `Owner`, `Name`, `APIBaseURL`, `Scopes`, `DisplayName`, `Aliases`, `CreatedAt`, and `UpdatedAt`; `RepositoryScope` is a constrained enum with `issues` and `wiki`; `RepositoryRegistry` exposes `AddRepository`, `GetRepository`, `ListRepositories`, `RepositoryStatus`, `ValidateAliasSet`, and `ValidateScope`. Add a durable repo registry storage contract at the cache schema/migration boundary: `repos(repo_id PRIMARY KEY, owner, name, api_base_url, scopes, display_name, created_at, updated_at)` and `repo_aliases(alias PRIMARY KEY, repo_id REFERENCES repos(repo_id), created_at)`, so repository aliases are globally unique while record aliases remain a separate repo-scoped concept in Task 2. Add CLI repo command surface `repo add` and `repo status --repo <repo_id>` that calls `RepositoryRegistry`; `repo add` validates non-empty stable `repo_id`, normalized owner/name/API base URL, non-empty enabled scopes, duplicate scopes collapsed, no duplicate `repo_id`, and no cross-repo repository-alias collision. `repo status` returns public-safe fields: `repo_id`, owner/name, redacted or sanitized API base URL without credentials, enabled scopes, display name, repository aliases, binding state, alias conflict state, cache availability handoff, index availability handoff, and failure class; cache/index fields are handoff diagnostics supplied by cache/index consumers and may be `unknown`, `ready`, `cache_empty`, `missing_index`, or `stale_index`. Decommission-6 is enforced by keeping repository aliases out of any global record-resolution product path: a repo alias selects a repository only, never a source record, and duplicate repository aliases fail at bind time with typed conflict and no partial row.
Acceptance Criteria: A developer runs `gitcode-mcp repo add --repo fixture-a --owner owner-a --name repo-a --api-base-url https://example.invalid/api --scopes issues,wiki --alias proj` and then `gitcode-mcp repo status --repo fixture-a`; the CLI reports configured metadata, enabled scopes, repository aliases, public-safe API base URL, `binding_state: ready`, and cache/index handoff fields without token or private-path disclosure. A developer then runs `repo add` with an existing `repo_id` or colliding repository alias and receives a typed conflict, nonzero exit status, and no partial repo or alias row. A developer also runs `repo status --repo missing-repo` and receives a typed not-found diagnostic with actionable failure class. Executable evidence is a CLI integration test using temporary config/cache state with two sanitized fixture repositories and one ready status plus one diagnostic failure state.
Workload: 1.5 MM

### Task 2: Repo Scope Resolver Change
Outcome IDs: outcome-2, outcome-6, outcome-11
Outcome Role: primary_product
Decommission IDs: decommission-6
Change Type: change
Description: Change identity routing so every repository-bound read, write, sync, MCP, and snapshot selection receives an explicit `repo_id` and filters through repository scope. The component-local resolver owns precise alias-class rules for stable source ids, record aliases such as `issue:42`, remote ids, and unscoped lookup behavior. It hands normalized repository metadata to other components while leaving their cache queries, transports, adapter calls, and snapshot internals owned elsewhere.
Existing Behavior / Reuse: Current service request structs for search/get/list/backlinks/snippet/sync status/export/diff do not carry `repo_id`, and current MCP tool schemas do not require `repo_id` on read tools. Current cache schema has unscoped `sources`, `identity_map`, `remote_revisions`, `sync_events`, `chunks`, and conflicts keyed by `source_id` or global alias values; current `ResolveAlias` can resolve an alias without repository scope. Reuse current `Source`, `Identity`, `RemoteAlias`, source summary/rendering, service request dispatch, MCP JSON-RPC handling, and snapshot request patterns, but replace unscoped alias resolution as a product runtime path. Existing legacy unscoped lookup may remain only as an internal migration/test helper with a name that cannot be called by CLI/MCP product commands.
Detailed Design: Choose the explicit identity invariant: all source and record identity keys are `(repo_id, source_id)` scoped, with mandatory `repo_id` columns on repository-bound cache rows and no globally meaningful source id for product routing. The storage contract becomes: `sources` uniqueness is `(repo_id, id)` with `repo_id REFERENCES repos(repo_id)`; `identity_map` primary uniqueness is `(repo_id, alias_type, alias)` plus `UNIQUE(repo_id, source_id, alias_type, alias)`; remote aliases are unique by `(repo_id, remote_type, remote_id)`; snapshots are unique by `(repo_id, snapshot_id)` and snapshot source/chunk selections include `repo_id`; related records such as chunks, links, sync events, remote revisions, conflicts, and audit handoffs carry `repo_id` so consumers can enforce the same boundary. Add `RepoScopedAliasResolver` with `ResolveSourceID(repo_id, source_id)`, `ResolveAlias(repo_id, alias_type, alias)`, `ResolveRemote(repo_id, remote_type, remote_id)`, `ResolveRecordRef(repo_id, ref)`, and `DiagnoseUnscopedAlias(alias_type, alias)`; every method first validates that the repository exists through `RepositoryRegistry`. Alias class rules are: repository aliases are globally unique and resolve only to `repo_id`; stable source ids are scoped as `(repo_id, source_id)` and may duplicate across repositories; record aliases such as `issue:42`, `wiki:Home`, and local path aliases are scoped as `(repo_id, alias_type, alias)` and may duplicate across repositories; remote ids are scoped as `(repo_id, remote_type, remote_id)` and never replace stable source ids; unscoped product lookup returns `ambiguous_alias` when more than one repo contains the alias, `repo_required` when a repo-aware product surface omits `repo_id`, or typed not-found when no scoped match exists. Add a `RepositoryRoute` handoff object with `RepoID`, `Owner`, `Name`, `APIBaseURL`, `Scopes`, and resolved source identity; service request contracts and MCP schema handoff add `repo_id` to repository-bound operations, while CLI parsing treats `--repo <repo_id>` as the repository scope rather than owner/name. Sync/write routing owned by repo-binding is `BuildAdapterRoute(repo_id, requested_scope)`: validate repo exists, validate requested `issues` or `wiki` scope is enabled, return owner/name/API base URL to adapter callers, and reject disabled-scope operations before adapter construction or network calls; this is a handoff contract for cache_sync, gitcode_adapter, and cli_write, not an adapter implementation. Decommission-6 is enforced by deleting unscoped alias resolution from CLI/MCP product control flow, rejecting or diagnosing omitted `repo_id`, and proving same alias values in different repos never cross-resolve.
Acceptance Criteria: After two fixture repositories are added and both contain record alias `issue:42`, a developer runs a CLI read with `--repo fixture-a issue:42` and receives only fixture-a data with response metadata containing `repo_id: fixture-a`; the same command with `--repo fixture-b issue:42` returns only fixture-b data. A developer runs an unscoped product read for `issue:42` and receives `ambiguous_alias` or `repo_required` instead of the first matching source, while a scoped non-colliding lookup such as `--repo fixture-a wiki:Home` resolves successfully. An MCP JSON-RPC read tool call with `repo_id: fixture-a` returns the same scoped source as the equivalent CLI read, and snapshot/export selection rejects a source alias from another repo with typed not-found or validation-failed. A developer also attempts sync/write routing for a repo whose `wiki` scope is disabled; the repo-binding route rejects the operation before any live adapter call, proven by a fixture or stubbed-adapter test that records zero adapter invocations.
Workload: 2.5 MM

## Cross-Cutting Constraints
- Every repository-bound cache row and read response carries `repo_id` — repository binding is the scope boundary consumed by cache, CLI, MCP, and snapshot components.
- Remote ids are aliases, not replacements for stable source ids — repo binding scopes remote ids without promoting issue numbers or wiki slugs to global identity.
- Routine reads remain cache-first — repository binding must route to local records and must not trigger live GitCode network access.
- Sync and write operations must validate repository scope before adapter handoff — disabled issue/wiki scopes fail locally before any live network call.
- Public-safe diagnostics are mandatory — repo status and alias errors must not expose credentials, private paths, cookies, or unsanitized remote payloads.

## Data And Control Flow
- `repo add` validates repository metadata — CLI actor -> `RepositoryRegistry` -> durable repo and repo-alias rows — duplicate `repo_id` or repository alias returns conflict before any cache record mutation.
- `repo status` assembles diagnostics — CLI actor -> `RepositoryRegistry` -> cache/index handoff probes — output includes configured metadata, enabled scopes, repository aliases, alias conflict state, cache/index availability, and public-safe failure class.
- Repo-scoped read resolves identity — CLI/MCP actor -> `RepoScopedAliasResolver` -> cache source identity `(repo_id, source_id)` — `repo_id` is validated before alias lookup and included in response metadata.
- Collision handling rejects unscoped product aliases — CLI/MCP actor -> alias resolver diagnostic — colliding aliases return `ambiguous_alias` and omitted repo scope returns `repo_required` for repo-aware surfaces.
- Sync/write route validates enabled scope — sync/write caller -> repository-binding `BuildAdapterRoute` -> adapter handoff — missing repo or disabled issue/wiki scope fails before adapter construction or network calls.
- Snapshot selection uses repository scope — snapshot consumer -> repository binding -> snapshot/cache component — aliases from another repo are not eligible for export or diff lookup.

## Component Interactions
- `repo-binding` -> `cache_sync` — supplies repo registry storage contract, `(repo_id, source_id)` identity invariant, repo-scoped alias constraints, and repository route validation for sync inputs.
- `repo-binding` -> `cli_read` — defines `--repo <repo_id>` validation, repository status fields, scoped alias diagnostics, and response metadata expectations for cache-first read commands.
- `repo-binding` -> `mcp_server` — defines required `repo_id` argument semantics for repo-aware MCP tools, ambiguous-alias errors, and schema handoff for scoped source resolution.
- `repo-binding` -> `gitcode_adapter` — provides owner/name/API base URL and enabled-scope route only after local validation; adapter owns live HTTP behavior.
- `repo-binding` -> `cli_write` — validates repo existence and requested issue/wiki scope before write command handoff, preventing disabled-scope live calls.
- `repo-binding` -> `snapshot_diff` — provides repo-scoped source, alias, and snapshot selection so export/diff never resolves cross-repo aliases.

## Rationale
The current runtime already has source, identity, CLI, MCP, and cache concepts, but repository identity is missing as a first-class product boundary. Adding a repository registry and scoped alias resolver is the smallest component-local change that satisfies the approved architecture and replaces unscoped aliases.
