# Component Design: Milestone Adapter

## Summary
The milestone adapter is a missing live-provider capability: existing source concepts cover live issues, comments, wiki, search, issue/wiki writes, endpoint helpers, cache records, sync staging, and diagnostics, but no milestone model, endpoint, provider method, sync caller, or adapter test exists. This design adds live `/api/v5` milestone list/get decoding and cache-record projection for `kind=milestone`.

## Top-Level Alignment
This component implements the approved milestone decision: milestones are supported, not deferred, through documented `/api/v5/repos/{owner}/{repo}/milestones` and `/milestones/{id}` routes. It also participates in the route/schema coverage contract and error taxonomy by only executing when `RouteSchemaMatrix` marks milestones `supported` with `OpenAPI` evidence, and by producing `schema_decode` diagnostics for malformed milestone payloads.

## Tasks

### Task 1: Add milestone live decoder
Outcome IDs: outcome-4, outcome-1, outcome-7, outcome-10
Outcome Role: supporting_evidence
Decommission IDs: decommission-4
Change Type: add
Description: Add a milestone-specific live adapter path that lists and gets GitCode milestones through the documented `/api/v5` milestone routes. The component owns milestone endpoint selection, response models, schema validation, normalization into cache-ready milestone records, and adapter tests that exercise production code paths with `httptest` responses. Malformed milestone response bodies must be classified as `schema_decode`, while HTTP 4xx responses must be classified as `api_validation`, not transport or configuration failures.
Existing Behavior / Reuse: Reuse the existing `internal/gitcode` live provider boundary, HTTP client request execution, endpoint-helper pattern, repository owner/name validation, issue-style paged JSON read pattern, cache `Record` projection shape, sync service staging/commit handoff, and diagnostics classifier/error-code vocabulary. Source inspection confirms milestone support is absent today: there is no milestone endpoint helper, `Milestone` model, provider-facing `ListMilestones` or `GetMilestone` method, sync milestone caller, cache projection for `kind=milestone`, or adapter-level milestone test. Cache schema ownership remains with the cache component; sync scheduling and record commit ownership remain with the sync service; error class definitions remain owned by the diagnostics/error-classifier component, while this adapter supplies typed API/decode failures to those existing owners.
Detailed Design: Add component-local milestone route helpers equivalent to the existing issue endpoint helpers: `milestonesEndpoint(owner, repo)` builds `GET /api/v5/repos/{owner}/{repo}/milestones`, and `milestoneEndpoint(owner, repo, id)` builds `GET /api/v5/repos/{owner}/{repo}/milestones/{id}` with the same path escaping rules as existing live routes. Add provider-facing live methods `ListMilestones(ctx, MilestoneListRequest)` and `GetMilestone(ctx, MilestoneRequest)` on the GitCode live provider/client surface; the list method first checks the construction-time `RouteSchemaMatrix` entry for milestones and proceeds only when status is `supported`, route family is `/api/v5`, and evidence class is `OpenAPI`, otherwise returning the matrix diagnostic without issuing an HTTP request. The get method performs the same matrix check and then executes a single-object JSON request for the configured milestone id; the list method uses the existing issue-style paged request convention with `page` and `per_page` query parameters when the shared helper supports pagination, and decodes each page as a top-level JSON array of milestone objects, not an envelope object.

Define `MilestoneListRequest` with repository owner/name, page/per-page options if already used by the live client, and provider mode metadata; define `MilestoneRequest` with owner/name and milestone id. Define a live response model `Milestone` with cache-projection fields: `RemoteID string`, `SourceID string`, `Title string`, `Body string`, `Status string`, `DueOn string`, `CreatedAt string`, `UpdatedAt string`, and optional `HTMLURL string`. The documented display-name field is `title`; `title` is required, must be a non-empty JSON string, and there is no `name` fallback. If a response contains `name` but omits `title`, the decoder emits `schema_decode` with field path `milestone.title`, expected `non-empty string`, and received `missing`.

Milestone decode contract: list response is a top-level JSON array of milestone objects; get response is one JSON object; unknown fields are ignored. Required fields are `id` and `title`; `id` accepts a JSON number with integer value or a non-empty decimal string and normalizes to decimal `remote_id`; missing, fractional, negative, object, array, boolean, or non-decimal string id emits `schema_decode` at `milestone.id`. `source_id` is derived as `MILESTONE-<remote_id>`; the get route id and response id are not required to match for transport success, but a mismatch is surfaced as a `schema_decode` diagnostic at `milestone.id` because the cache identity would be ambiguous. Optional text fields `description` and `body` are accepted as strings; if both are present, `description` wins for cache body; absent values default to `""`; non-string values emit `schema_decode` at the offending field. Optional status fields accept `state` first, then `status`; values are lower-cased and mapped so `open` and `active` become cache status `open`, `closed` becomes `closed`, absent value defaults to `open`, and non-string or unrecognized string values emit `schema_decode` at `milestone.state` or `milestone.status`. Optional date/timestamp fields `due_on`, `due_date`, `created_at`, and `updated_at` must be strings when present; due fields accept `YYYY-MM-DD` or RFC3339 and normalize to `YYYY-MM-DD`, timestamps accept RFC3339 and preserve canonical RFC3339 text, absent values default to `""`, and unparsable strings emit `schema_decode` with the field path and expected format.

Add a milestone normalizer `decodeMilestone` that converts raw response objects into `Milestone` values and accumulates field-path diagnostics without silently dropping malformed objects. A mixed list with any invalid element fails the list call with `schema_decode` naming the array index, such as `milestones[2].title`, rather than partially succeeding. Add a cache projection function owned by this adapter, such as `Milestone.ToRecord(provenance)`, that returns cache-ready records with `kind=milestone`, `remote_id`, `source_id`, `title`, `body`, `status`, timestamp fields when supported by the cache record model, and provenance supplied by the sync/cache caller; the adapter does not commit records directly. Add a sync-facing milestone caller hook that lets the sync service request live milestones and receive cache-ready projections, while record commit, provenance assignment, and cache read exposure remain owned by sync/cache components.

Route all milestone HTTP status responses through the existing live response classification boundary: 400, 401, 403, 404, 409, and 422 become `api_validation` with status and sanitized response message; 5xx and network errors remain `live_transport_failure`; absent credentials remain `config_credential`; malformed JSON, wrong top-level list/object shape, missing required fields, type mismatches, id mismatch, malformed title, malformed status, and malformed dates become `schema_decode`. For `decommission-4`, replace the old ambiguity for milestone paths by ensuring schema/decode/API failures never use the network-unavailable or configuration-failure sentinel; adapter tests enforce this negative invariant by failing if 400, malformed JSON, or schema mismatch classifies as `live_transport_failure` or credential/configuration failure.
Acceptance Criteria: Developer runs `go test ./...` and `git diff --check`; milestone adapter tests drive the production live provider against stubbed external GitCode routes `GET /api/v5/repos/{owner}/{repo}/milestones` and `GET /api/v5/repos/{owner}/{repo}/milestones/{id}` with the `RouteSchemaMatrix` milestone entry set to `supported` and `OpenAPI`, proving the supported milestone path reaches live adapter list/get code rather than a document-only matrix. The tests observe cache-ready records with `kind=milestone`, `source_id=MILESTONE-<id>`, normalized `remote_id`, required `title`, body from `description`, mapped status, parsed dates/timestamps, and no real network, credentials, SSH agent, or Keychain access. Negative tests cover malformed JSON, wrong list envelope/object shape, missing or malformed identity, missing `title` with only `name` present, wrong-type or empty `title`, malformed date/status values, mixed valid/invalid list entries, get/list id mismatch, and HTTP 400; all schema/body failures produce `schema_decode`, while mocked HTTP 400 produces `api_validation` and never `live_transport_failure` or configuration failure.

## Cross-Cutting Constraints
- Offline validation only — milestone tests must use stubbed external GitCode routes so normal `go test ./...` needs no credentials, network, SSH agent, or Keychain
- Four-class error taxonomy — milestone HTTP and decode failures must align with `api_validation`, `schema_decode`, `config_credential`, and `live_transport_failure` boundaries
- Cache-first product read path — milestone records are projected into cache-ready `kind=milestone` records so CLI/MCP reads can remain cache-first
- Route/schema coverage contract — milestone list/get execution is gated by `RouteSchemaMatrix` support status and OpenAPI evidence, with tests proving runtime code-path coverage

## Data And Control Flow
- CLI or sync actor requests milestone sync/read — sync invokes the live milestone adapter only after provider mode resolves to live and `RouteSchemaMatrix` marks milestones `supported` with `/api/v5` and `OpenAPI` evidence
- Live milestone adapter issues GitCode requests — `GET /api/v5/repos/{owner}/{repo}/milestones` returns a paged top-level array and `GET /api/v5/repos/{owner}/{repo}/milestones/{id}` returns one object
- GitCode response enters milestone decoder — decoder validates required `id` and `title`, optional state/date/text fields, and unknown-field tolerance before returning normalized milestone data or `schema_decode`
- Normalized milestone enters cache projection — adapter emits cache-ready `kind=milestone` records, while sync/cache ownership supplies provenance and commits records
- Failure exits through classifier — HTTP 4xx maps to `api_validation`; malformed JSON and shape mismatch map to `schema_decode`; transport and credential failures keep their separate classes

## Component Interactions
- `RouteSchemaMatrix` -> `milestone-adapter` — milestone adapter reads the construction-time milestone entry and executes list/get only when status is `supported`, route family is `/api/v5`, and evidence class is `OpenAPI`
- `milestone-adapter` -> `live_adapter` — exposes list/get milestone methods and endpoint construction for `/api/v5/repos/{owner}/{repo}/milestones` routes through the existing GitCode live provider/client boundary
- `milestone-adapter` -> `sync_service` — returns normalized milestone values or cache-ready projections; sync owns scheduling, provenance assignment, and cache commit
- `milestone-adapter` -> `cache` — emits records with `kind=milestone`, stable source id, remote id, title, body, status, timestamps, and caller-supplied provenance without taking ownership of cache schema migration
- `milestone-adapter` -> `error_classifier` — supplies typed decode/API failures so classifier preserves `schema_decode` versus `api_validation` boundaries
- `test_suite` -> `milestone-adapter` — exercises RouteSchemaMatrix-supported list/get success, schema diagnostics, and HTTP 400 classification with `httptest` external routes under `go test ./...`

## Rationale
The approved architecture explicitly implements milestones in this iteration, and the configured Component Impact marks the milestone adapter as detailed. Because the repository currently has no milestone-specific model, route, provider method, sync path, cache projection, or tests, a component-local live decoder and cache projection task is required.
