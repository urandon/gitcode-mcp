# Component Design: GitCode Adapter

## Summary
The `gitcode-adapter` component is affected because the approved architecture requires a new `internal/gitcode` package that owns all GitCode `/api/v5` HTTP access, typed degradation errors, endpoint templates, pagination, attachment handling, and explicit write behavior. The repository currently has no `internal/gitcode` package, so the component-local work is additive.

## Top-Level Alignment
`internal/gitcode` sits below `internal/service` as the only package that crosses the network boundary to GitCode. It never imports `cache` or `service`, exposes context-aware adapter contracts, and returns typed package errors so the service layer can preserve cache-first safety guarantees.

## Tasks

### Task 1: Add Client and error types
Outcome IDs: outcome-3, outcome-13
Outcome Role: primary_product
Decommission IDs: none
Change Type: add
Description: Add the `internal/gitcode` adapter package with the component-local `Client` interface, HTTP client implementation, response models, attachment models, retry executor, and typed error family. This establishes the read-side adapter boundary for issues, wiki pages, comments, attachments, and search while keeping cache and service concerns outside the package. The same error types encode poor-network and degraded-response behavior required by sync callers.
Existing Behavior / Reuse: The required functionality is confirmed absent: `internal/gitcode` does not exist, and the current repository only has `internal/cli` plus `cmd/gitcode-mcp` scaffolding. Reuse the standard library `context`, `net/http`, `encoding/json`, `io`, `net/url`, and `time`; do not introduce an external HTTP client.
Detailed Design: Add a `Client` interface with context-aware read methods `ListIssues`, `GetIssue`, `ListIssueComments`, `GetWikiPage`, `ListWikiPages`, `Search`, `ListIssueAttachments`, and `GetAttachment`. Add package-owned request/response structs including `Issue`, `IssueSummary`, `Comment`, `WikiPage`, `SearchResult`, `AttachmentSummary`, `AttachmentBody`, `AttachmentRequest`, and `Page[T]`; `AttachmentBody` carries `ID`, `Name`, `ContentType`, `Size`, `Body`, source endpoint, and checksum when present. Add an `HTTPClient` implementation configured by `Config{BaseURL, Token, Timeout, MaxResponseSize, MaxRetries, UserAgent, Pagination}`; bearer auth is applied only when a token is provided, and token values must never be copied into returned errors.

The shared request executor builds a request through endpoint builders, applies the caller context deadline, sends it through `http.Client`, bounds the response with `MaxResponseSize`, and decodes JSON into package models or returns typed errors. Read retry behavior is bounded and deterministic: `MaxRetries` means additional attempts after the first request, retryable statuses are `429` and transient `5xx`, retryable transport failures are DNS/connection reset/temporary network errors, context cancellation or deadline expiry stops immediately, and all attempts for a single logical read use the same caller context. For `429`, parse `Retry-After` as seconds or HTTP-date; sleep until the earlier of `Retry-After` or context cancellation, retry while attempts remain, and after retry exhaustion return `ErrRateLimited{RetryAfter, RawRetryAfter, Endpoint, Attempts}` with no successful records. For list/search pagination, page results are accumulated privately and are returned only after every requested page succeeds; if any page exhausts retries, decodes partially, exceeds size, or sees context cancellation, the method returns the typed error and a nil/empty result, never a partial page/list success.

Status/error mapping is explicit: `401` -> `ErrAuthExpired`; `403` -> `ErrForbidden{Endpoint, Status, Message, Recovery}` with non-retryable permission guidance that does not suggest network retry; `404` -> `ErrNotFound` for ordinary lookups and `ErrRemoteNotFound` when the caller marks the request as a known remote alias sync; `409` -> `ErrConflict` for write routes; retry-exhausted `429` -> `ErrRateLimited`; retry-exhausted transient `5xx` or retryable transport errors -> `ErrNetworkUnavailable`; context deadline expiry -> `ErrNetworkUnavailable` with retry guidance; malformed JSON after a successful status -> `ErrPartialResponse`; truncated JSON or early EOF -> `ErrPartialResponse`; body larger than `MaxResponseSize` -> `ErrPayloadTooLarge`. Attachment fetches enforce `MaxResponseSize` before buffering the body, return `ErrPayloadTooLarge` for oversized bodies, and return `ErrPartialResponse` for truncated attachment metadata or content responses.

Define typed errors `ErrNetworkUnavailable`, `ErrRateLimited{RetryAfter, RawRetryAfter, Endpoint, Attempts}`, `ErrAuthExpired`, `ErrForbidden`, `ErrNotFound`, `ErrConflict`, `ErrPartialResponse`, `ErrRemoteCollision`, `ErrRemoteNotFound`, and `ErrPayloadTooLarge`, each implementing `error` and exposing stable fields for callers to inspect. `ErrRemoteCollision` is defined in this package only to share the cross-component error contract; normal identity-map alias collisions are detected and raised by service/cache reconciliation, while the adapter raises it only if a single remote response is internally contradictory. The invariant is that this package never reads or writes cache state and never returns partially decoded records as success.
Acceptance Criteria: A developer triggers adapter contract tests with `go test ./internal/gitcode/... -run TestContract`; the target product API is `Client.GetIssue` against an `httptest.Server` serving sanitized fixture JSON, and the expected response is a structured `Issue` with `id`, `title`, `body`, `status`, `labels`, `created_at`, and `updated_at` matching the fixture. A developer triggers attachment coverage with `go test ./internal/gitcode/... -run TestAttachmentContract`; the target product APIs are `Client.ListIssueAttachments` and `Client.GetAttachment` against sanitized fixture responses, and the expected result is attachment metadata plus bounded attachment content, with an oversized fixture returning typed `ErrPayloadTooLarge`. A developer triggers retry coverage with `go test ./internal/gitcode/... -run TestReadRetry`; the target routes include one read route returning `429` then `200` under the same context and producing the successful record, and one list route exhausting `429` retries and returning `ErrRateLimited{RetryAfter}` with no records. A developer triggers timeout and failure coverage with `go test ./internal/gitcode/... -run 'TestTimeout|TestFailureModes'`; routes exercise 401, 403, 404, 409, 429 with valid and invalid `Retry-After`, 5xx/network failures, malformed JSON, truncated JSON, and max-size truncation, with 403 returning `ErrForbidden` stable fields and non-retry permission guidance, network timeout returning `ErrNetworkUnavailable` containing endpoint context and retry guidance, and rate limit returning `ErrRateLimited.RetryAfter`.

### Task 2: Add endpoints and pagination
Outcome IDs: outcome-3
Outcome Role: supporting_evidence
Decommission IDs: none
Change Type: add
Description: Add centralized endpoint construction and pagination strategy objects inside `internal/gitcode`. This isolates undocumented or unstable GitCode `/api/v5` URL and pagination behavior from the rest of the adapter. The implementation keeps the `Client` method signatures stable while allowing page/per-page pagination to be replaced by cursor pagination later.
Existing Behavior / Reuse: No endpoint builder, attachment endpoint builder, write endpoint builder, or pagination code exists today. Reuse the `HTTPClient` request path from the client task and keep endpoint construction package-local rather than spreading URL string formatting across methods.
Detailed Design: Add endpoint builder functions for read routes: `listIssuesEndpoint(owner, repo)`, `getIssueEndpoint(owner, repo, number)`, `listIssueCommentsEndpoint(owner, repo, number)`, `getWikiPageEndpoint(owner, repo, slug)`, `listWikiPagesEndpoint(owner, repo)`, and `searchIssuesEndpoint()`. Add package-local attachment endpoint builders for the discovered path shape behind named functions such as `issueAttachmentsEndpoint(owner, repo, number)` and `attachmentEndpoint(owner, repo, attachmentID)`; because public GitCode docs do not confirm attachment path shape, these builders are the single isolated update point when sanitized fixtures confirm the final route. Endpoint builders URL-escape path segments and query parameters, preserve `/api/v5`, and remain independent of auth, idempotency, and cache state.

Add centralized write endpoint builders for every write method in this component: `createIssueEndpoint(owner, repo)`, `updateIssueEndpoint(owner, repo, number)`, `createIssueCommentEndpoint(owner, repo, number)`, `createWikiPageEndpoint(owner, repo)`, `updateWikiPageEndpoint(owner, repo, slug)`, `addLabelEndpoint(owner, repo, number)`, and `removeLabelEndpoint(owner, repo, number, label)`. Keep uncertain GitCode write path shapes isolated behind these named builders until fixture capture confirms whether labels and wiki mutation routes use issue-number, label-name, or request-body targeting. Write methods in Task 3 must call these builders rather than constructing URLs inline.

Add a `PaginationStrategy` interface with methods that apply request query parameters and parse next-page state from HTTP headers or response envelopes. Provide a default `PagePagination` implementation using `page` and `per_page`, plus a test cursor strategy that proves pagination can be swapped without changing `Client` signatures. List/search methods loop through pages only through `PaginationStrategy`, stop on context cancellation, and return a typed error without publishing partially decoded records as a successful result if any page fails.
Acceptance Criteria: A developer triggers endpoint tests with `go test ./internal/gitcode/... -run TestEndpointsTemplate`; the target API is the package endpoint builder, and `GetIssue` for owner `example-owner`, repo `example-repo`, number `42` visibly produces `/api/v5/repos/example-owner/example-repo/issues/42`. A developer triggers attachment endpoint tests with `go test ./internal/gitcode/... -run TestAttachmentEndpointsTemplate`; the target API is the package attachment endpoint builder, and the expected route is generated from sanitized fixture-backed expectations while `Client.ListIssueAttachments` and `Client.GetAttachment` signatures remain unchanged. A developer triggers write endpoint tests with `go test ./internal/gitcode/... -run TestWriteEndpointsTemplate`; the target APIs are centralized builders for create/update issue, create/update wiki page, create comment, add label, and remove label, and the expected outcome is that every write method route is generated by a named builder with no inline URL construction. A developer triggers pagination tests with `go test ./internal/gitcode/... -run TestPaginationSwappable`; the target API is `Client.ListIssues` against an `httptest.Server` with two pages, and replacing `PagePagination` with a test cursor strategy changes only page traversal behavior, not returned records or public adapter method signatures.

### Task 3: Add idempotent writes
Outcome IDs: outcome-12
Outcome Role: primary_product
Decommission IDs: none
Change Type: add
Description: Add explicit GitCode write methods to `internal/gitcode` for issue, wiki, comment, and label mutations. These methods are adapter-owned network operations and remain callable only by higher-level explicit command paths, not routine reads. Conflict and idempotency behavior is represented through package request headers, response models, and typed errors.
Existing Behavior / Reuse: Write behavior does not exist in the current scaffold. Reuse the `HTTPClient` request executor, endpoint builders, JSON encoder/decoder, auth handling, retry policy, pagination-independent request construction, and typed errors from the read adapter work.
Detailed Design: Extend the package API with write request/response types and methods `CreateIssue`, `UpdateIssue`, `CreateIssueComment`, `CreateWikiPage`, `UpdateWikiPage`, `AddLabel`, and `RemoveLabel` as confirmed by GitCode fixtures. Add `WriteOptions{IdempotencyKey, IdempotencyNonce}` and an `IdempotencyKey` generator that hashes stable serialization of method name, target, payload, and either a caller-provided nonce or creation timestamp, then truncates to the configured prefix length. The idempotency key is computed once per logical write call before the first HTTP attempt; all retry attempts for that operation reuse the exact same key, and caller-provided `WriteOptions.IdempotencyKey` is the deterministic replay mechanism for tests and resumed operations.

Each write method calls the centralized write endpoint builder from Task 2, sets the `Idempotency-Key` header, sends JSON payloads, decodes 200/201 responses into package models, maps 409 to `ErrConflict{LocalPayload, RemotePayload}`, maps retry-exhausted 429 to `ErrRateLimited`, and never retries with a new key. Retried 429 or transient write attempts wait according to the parsed retry policy while preserving the original key and request payload; context cancellation stops retry and returns the context-derived typed error. The adapter performs no automatic conflict resolution and does not import or update `sync_events`; it returns the created/updated model, the idempotency key used, or structured error evidence for `internal/service` and `internal/cache` to log write-ahead state.
Acceptance Criteria: A developer triggers write tests with `go test ./internal/gitcode/... -run TestWriteIdempotency`; the target product API is `Client.CreateIssue` against an `httptest.Server`, and the captured HTTP request visibly includes an `Idempotency-Key` header and JSON payload generated through `createIssueEndpoint`. When the server returns 409 Conflict, the expected response is typed `ErrConflict` containing both the local request payload and remote response payload, with no automatic overwrite and no successful record returned. When the server returns 429 followed by 201, the expected result is a structured created issue and the server observes the exact same idempotency key on every retry attempt; when the caller supplies `WriteOptions.IdempotencyKey`, the server observes that exact key for deterministic replay. A developer triggers route coverage with `go test ./internal/gitcode/... -run TestWriteUsesEndpointBuilders`; the target APIs include `CreateIssue` plus at least one label or wiki write route, and the expected result is that requests pass through the centralized builders from Task 2.

## Cross-Cutting Constraints
- `internal/gitcode` must never import `internal/cache` or `internal/service` — the approved dependency direction keeps network access as an adapter detail below the service layer.
- Every adapter method accepts `context.Context` — deadlines and cancellation are required for timeout handling, retry bounding, and sync safety.
- Routine reads remain cache-first outside this package — live GitCode calls are initiated only by explicit service sync/write paths.
- Fixtures and errors must be public-safe — credentials, internal URLs, and non-public project identifiers must not appear in tests, logs, or returned error text.
- Write methods must carry idempotency keys and must not resolve conflicts automatically — this preserves explicit write safety and enables service/cache audit logging.
- Attachment handling must be bounded by `MaxResponseSize` — oversized or truncated bodies are degradation cases, not successful partial records.
- Read list/search methods must not return partial success after retry exhaustion — cache sync callers require all-or-error behavior to preserve cache safety guarantees.

## Data And Control Flow
- Service sync or write code calls an `internal/gitcode.Client` method with context and typed request data — `internal/gitcode` owns HTTP execution and returns package models, idempotency evidence, or typed errors only — callers decide whether and how to update cache state.
- `HTTPClient` builds the endpoint URL, applies auth and idempotency headers, sends the request, bounds and decodes the response, retries retryable failures within the caller context, and maps status/decode failures into typed errors — no cache mutation occurs in this package.
- List/search operations delegate page advancement to `PaginationStrategy` and keep accumulated records private until the operation fully succeeds — `Client` signatures remain stable even if GitCode pagination changes after fixture discovery.
- Attachment operations use endpoint builders isolated in the adapter — undocumented path changes are absorbed by endpoint and fixture updates without changing `ListIssueAttachments` or `GetAttachment`.
- Identity-map collision ownership is split deliberately — `internal/gitcode` defines `ErrRemoteCollision`, but service/cache reconciliation raises normal alias collisions after comparing remote aliases with the local identity map.
- Write operations compute or accept one idempotency key per logical operation — all retries reuse that same key, and 409 responses produce `ErrConflict` with local and remote payloads and stop without overwriting either side.

## Component Interactions
- `internal/service` -> `internal/gitcode` — service owns orchestration and calls `Client` for explicit sync/write operations; gitcode returns typed records, attachment bodies, idempotency evidence, and errors only.
- `internal/gitcode` -> `GitCode /api/v5` — adapter owns all HTTP methods, URL templates, auth header application, response parsing, retries, pagination, attachment size limits, and status-code mapping.
- `internal/gitcode` -> `fixtures/api/v5` — adapter contract tests consume sanitized issue, wiki, search, comment, label, write, and attachment fixtures through `httptest.Server`; production runtime never depends on fixture files.
- `internal/cli` -> `internal/service` -> `internal/gitcode` — explicit write CLI commands eventually reach adapter write methods, while MCP write tools remain deferred in the first slice.

## Rationale
The GitCode adapter is a new required component because the architecture assigns all `/api/v5` interoperability, error normalization, attachment handling, pagination, retry semantics, and idempotent write mechanics to `internal/gitcode`. Keeping this work isolated prevents cache and service types from leaking into transport code and lets fixture-backed tests validate remote behavior without live network access.
