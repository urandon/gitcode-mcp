# Component Design: Diagnostics

## Summary
Diagnostics is materially affected because iteration 4 requires stable, typed live-mode failure classes instead of generic sync/write/provider errors. The component-local work is to classify supplied errors and sentinel conditions into canonical redacted diagnostics while leaving detection, HTTP, credential lookup, cache inspection, and audit writes to their owning components.

## Top-Level Alignment
The approved architecture assigns `diagnostics` ownership of user-visible classes for missing credentials, live auth failure, transport/API failure, configuration errors, unsupported payloads, and fixture fallback detection. This design keeps that boundary narrow: `internal/diagnostics` owns canonical codes, deterministic precedence, redaction, and rendering; credential-resolution, repository-binding, live-provider, sync/write services, and integration tests own the inputs that prove those classes.

### Task 1: Classifier catalog change
Outcome IDs: outcome-2, outcome-4, outcome-6, outcome-8
Outcome Role: supporting_evidence
Decommission IDs: decommission-2, decommission-4
Change Type: change
Description: Change the diagnostics component around a typed live-readiness error catalog consumed by CLI startup, credential resolution, repository binding, live provider, sync/write services, and integration tests. The component-local role is to translate already-detected credential, provider, repository-binding, payload-validation, and fixture-route errors into one stable public failure class with redacted context. This task covers missing credentials, invalid selected API base URL, broader live configuration errors, unsupported mock/live payloads, live auth/transport/API failures, fixture read-only contexts, and fixture fallback sentinels.
Existing Behavior / Reuse: Reuse existing `internal/diagnostics` concepts for CLI error rendering and redaction if present, but confirm during implementation that no existing diagnostics API already provides the required canonical live-readiness failure classes, precedence, `http_attempted` invariants, and fixture-fallback sentinel classification. Reuse existing credential-resolution missing-token errors, repository-binding validation errors, live-provider HTTP/auth/transport errors, CLI text/JSON rendering surfaces, and mock/live payload validation errors as classifier inputs. Replace command-local generic errors such as `internal_error`, `invalid_query`, `write_unauthorized`, raw provider text, invalid-token fixture success, and live fixture read-only visibility with canonical diagnostics only when the caller supplies explicit live command context.
Detailed Design: Add or change diagnostics-owned types under `internal/diagnostics`: a `Code` string type, a `Diagnostic` value with `code`, `message`, `exit_class`, `http_attempted`, `retryable`, and redacted `context`, and a `Classifier` API that accepts an error plus command context. The command context should carry provider mode, command name, selected API base URL, repository binding identifier, cache path presence, audit path presence, payload kind, payload source, required identifier name, HTTP status when available, and whether live HTTP was attempted. Canonical codes are `missing_credential`, `live_auth_failure`, `live_transport_failure`, `live_api_failure`, `configuration_error`, `invalid_api_base_url`, `unsupported_mock_payload`, `fixture_read_only`, and `fixture_fallback_detected`; CLI text and JSON error output expose exactly one canonical `failure_class` for live-mode failures.

The classifier precedence must match the approved admission and degradation rules: broader missing or invalid live configuration first, then `missing_credential`, then `invalid_api_base_url`, then post-attempt `live_auth_failure`, `live_transport_failure`, `live_api_failure`, then `fixture_fallback_detected`. Broader `configuration_error` covers missing or invalid repository binding, missing cache path, invalid cache path, missing audit path for live write routes, invalid audit path, or other required live-mode settings and always renders with `http_attempted=false`. A syntactically invalid or explicitly rejected selected live base URL maps to `invalid_api_base_url` only after broader configuration and missing credential checks; therefore when credentials are missing and the selected base URL is invalid, the rendered class is `missing_credential` unless an earlier broader `configuration_error` applies.

Map no usable credential from credential-resolution to `missing_credential` with `http_attempted=false`. Map 401/403 live-provider responses to `live_auth_failure` only when `http_attempted=true`; if a 401/403-like provider error is supplied with `http_attempted=false`, classify it as a configuration or provider-input bug through the existing internal diagnostic path rather than rendering a post-HTTP live class. Map timeout, DNS, connection refused, or selected-endpoint reachability errors to `live_transport_failure` only when a live request was attempted, and map non-auth HTTP/API status or malformed API error responses to `live_api_failure` only when a live request was attempted. Pre-HTTP failures must never render post-HTTP classes, and transport/API failures require an attempted live request.

Map malformed or missing required issue/wiki/comment/create identifiers from mock/live payload validation to `unsupported_mock_payload`. Required context fields are `payload_kind`, `payload_source`, `required_field`, `command`, and `provider_mode`; all values pass through diagnostics redaction before rendering. `unsupported_mock_payload` can appear in user-facing CLI output for incompatible live API responses and in test output for malformed sanitized mock payloads, because the same adapter validation path rejects both.

Keep `fixture_read_only` available only for non-live fixture write contexts. In live mode, diagnostics must not scan cache/provider results for `ISSUE-42`, `WIKI-HOME`, or fixture-shaped success payloads; `sync-service`, `write-service`, or integration-test checks own that detection and hand diagnostics an explicit fixture-route error or fixture-fallback sentinel. When provider mode is live and the live route was expected, such supplied fixture-route errors or sentinels map to `fixture_fallback_detected`, replacing decommissioned invalid-token fixture success and `create-issue --live` fixture read-only routing. Fixture fallback in live mode overrides the raw fixture error only after live routing was expected; otherwise non-live fixture write behavior may render `fixture_read_only`.

Apply the diagnostics redaction filter to every message and context value, including tokens, cookies, authorization headers, full URLs with embedded credentials, cache/audit paths marked sensitive, and provider response bodies. Add diagnostics unit tests for every mapping and overlapping tie-break: broader configuration before missing credential, missing credential before invalid selected base URL, pre-HTTP errors not rendering post-HTTP classes, 401/403 requiring `http_attempted=true`, transport/API failures requiring attempted live request, unsupported payload context redaction, live fixture sentinel to `fixture_fallback_detected`, and non-live fixture read-only preserving `fixture_read_only`.

## Cross-Cutting Constraints
- Stable failure-class names are public CLI/API contract values — sync, write, doctor, and integration-test assertions depend on deterministic classes rather than Go error text.
- Diagnostics rendering must never expose tokens, cookies, authorization values, or secret-bearing URLs — credential and provider errors may carry context, but every output passes through redaction.
- Live mode is fail-closed — fixture fallback is classified as a live failure when supplied by sync/write/tests instead of being rendered as degraded fixture success.
- Diagnostics consumes detection inputs rather than producing them — credential-resolution, repository-binding, live-provider, sync/write services, and integration tests keep their architecture-owned responsibilities.

## Data And Control Flow
- Live command creates diagnostic context — `cli-startup` -> `diagnostics` — provider mode, command name, selected API base URL, repository binding metadata, cache/audit path presence, and HTTP-attempt state are supplied before rendering.
- Credential resolver failure is classified — `credential-resolution` -> `diagnostics` — no usable token maps to `missing_credential` before live HTTP is attempted and before selected-base-url classification.
- Repository validation failure is classified — `repository-binding` -> `diagnostics` — missing or invalid binding/cache/audit/live settings map to `configuration_error`, while invalid selected base URL maps to `invalid_api_base_url` only after broader configuration and missing-credential checks.
- Payload validation failure is classified — `live-provider` or mock payload validation -> `diagnostics` — malformed or missing required identifiers map to `unsupported_mock_payload` with redacted payload context.
- Provider HTTP failure is classified — `live-provider` -> `diagnostics` — 401/403 maps to `live_auth_failure`, transport failure maps to `live_transport_failure`, and other API failure maps to `live_api_failure` only when `http_attempted=true`.
- Fixture leakage sentinel is classified — `sync-service`, `write-service`, or `cli-integration-tests` -> `diagnostics` — these producers detect fixture identifiers, fixture-shaped success, or fixture read-only route errors; diagnostics only renders the supplied live-mode sentinel as `fixture_fallback_detected`.

## Component Interactions
- `credential-resolution` -> `diagnostics` — missing usable credential input is converted into `missing_credential` with `http_attempted=false`.
- `repository-binding` -> `diagnostics` — repository binding, cache path, audit path, and selected API base URL validation errors are converted into `configuration_error` or `invalid_api_base_url` according to classifier precedence.
- `live-provider` -> `diagnostics` — provider auth, transport, API, and response validation errors are converted into live-specific classes after request-attempt state is known.
- `sync-service` -> `diagnostics` — live sync fixture fallback sentinels supplied by the service are converted to `fixture_fallback_detected`; diagnostics does not inspect cache contents.
- `write-service` -> `diagnostics` — live create fixture read-only route errors or sentinels are converted to `fixture_fallback_detected` and block decommissioned fixture write fallback.
- `cli-integration-tests` -> `diagnostics` — tests assert stable failure classes and redacted messages through the real CLI product surface while owning fixture-identifier detection in test assertions.

## Rationale
Diagnostics is affected because iteration 4 acceptance depends on visible, typed distinctions between pre-HTTP credential/configuration failures, post-HTTP live-provider failures, malformed payloads, and forbidden fixture fallback. A shared classifier prevents divergent command-local strings while preserving the approved ownership boundary: other components detect conditions, and diagnostics classifies, redacts, and renders them.
