# Component Design: Sync Engine

## Summary
The sync engine resides in `internal/service/` (the package where `SyncToCache`, `stageIssue`, `stageWiki`, and `*Service` live in the actual source layout). Architecture component a7 assigns responsibility to `internal/sync/`, but the implementation reality is that all sync logic — provider dispatch, hash-based delta detection, retry, sync event recording — lives in `internal/service/`. This component design anchors all tasks to `internal/service/` types and adds three genuinely missing capabilities: multi-resource partial-failure aggregation via `SyncResources` + `PartialSyncError`, `StartedAt`/`CompletedAt` timestamps on `SyncEvent`, and `ZeroDelta` detection with always-persisted sync events.

## Top-Level Alignment
The approved top-level architecture assigns sync engine (a7) to dispatch to the selected provider, handle partial failure collection, produce sync events with start/end timestamps, and compute deltas without duplicates. Decommission-4 requires fixture-only sync to become one branch of a provider-dispatched design. Outcome-3 validates live sync, partial failure, and delta detection. The existing engine in `internal/service/` already has provider dispatch (`*Service.client`), hash-based delta (`contentHash` in `stageIssue`/`stageWiki`), retry (`fetchAndStage`), and `RemoteRevision` on `SyncEvent`. This design adds `SyncResources` for partial-failure collection, `StartedAt`/`CompletedAt` timestamps, and `ZeroDelta` with persistent events. Note on component ID: the architecture spec uses `internal/sync/` but the actual implementation puts sync logic in `internal/service/`; this design uses `internal-sync` as the catalog component ID while anchoring all tasks to `internal/service/` source concepts.

## Tasks

### Task 1: Add SyncResources for partial-failure collection
Outcome IDs: outcome-3
Outcome Role: primary_product
Decommission IDs: decommission-4
Change Type: add
Description: The sync engine currently has `SyncToCache` which syncs one resource per call. The architecture requires independent multi-resource fetching with success/failure aggregation and committed-successful semantics. `SyncResources` centralizes partial-failure aggregation engine-side, enabling future per-resource concurrency (goroutine-per-resource with a worker pool), consistent error wrapping via `PartialSyncError` that carries structured per-resource failure details (not just concatenated error strings), and engine-internal state tracking (success/failure counters updated atomically) that the CLI cannot replicate by iterating over `SyncToCache` — the CLI has no access to cache-commit ordering or in-flight resource state needed for safe concurrent dispatch.
Existing Behavior / Reuse: `SyncToCache` (service.go:753-838) already syncs one resource atomically — resolves idempotency key, records sync events, fetches via `fetchAndStage`, computes counts, and persists via `UpsertSyncGraph`. The CLI scoped sync mode (cli.go:518-540) iterates over `--issues`/`--wiki` flags calling `SyncToCache` per resource sequentially and collects errors, but this logic lives in `cmd/gitcode-mcp/` and lacks aggregated result structure or `PartialSyncError`. `SyncResources` brings this pattern into the sync engine with structured error aggregation and concurrency readiness.
Detailed Design: Add `SyncResources(ctx context.Context, reqs []SyncRequest) (*SyncResourcesResult, error)` to `*Service` (service.go). The method iterates over each `SyncRequest`, calls `SyncToCache(ctx, req)` for each independently, collects `SyncResult` on success and `error` on failure. On any failure, the method continues processing remaining resources (does not short-circuit). After all resources are processed, if any failures occurred, it returns `(*SyncResourcesResult, *PartialSyncError)` where `PartialSyncError` implements `error` and contains `Errors []ResourceError` with per-resource `SourceID`, `RemoteType`, and error details. `SyncResourcesResult` has `Results []SyncResult`, `SuccessCount int`, `FailureCount int`, and `Failures []ResourceError`. Successful `SyncToCache` calls have already committed to cache (records written, sync events persisted) by the time the aggregated result is returned — inherent to `SyncToCache`'s commit-before-return semantics. No partial rollback is needed since each `SyncToCache` is an independent atomic unit. Add `PartialSyncError` struct in service/types.go implementing `error` with an `Error() string` method that produces a summary including `SuccessCount` and `FailureCount`. Add `ResourceError` struct with `SourceID string`, `RemoteType RemoteType`, `Err error` fields. Add `SyncResourcesResult` struct with `Results []SyncResult`, `SuccessCount int`, `FailureCount int`, `Failures []ResourceError`. No changes to existing `SyncToCache` signature or behavior.
Acceptance Criteria: A Go test `go test -run TestSyncResourcesPartialFailure -count=1 ./internal/service/` constructs `*Service` via `NewWithClient` with a mock `gitcode.Client`, sets `GetIssue` to succeed for one source and fail with `ErrNotFound` for another, calls `SyncResources` with two `SyncRequest` values, and verifies the returned `SyncResourcesResult` has `SuccessCount == 1`, `FailureCount == 1`, `Results[0].Counts.Fetched > 0`, `Failures[0].SourceID` matches the failed source, and a `PartialSyncError` is returned; also verifies the successful resource's records are committed to the cache store via `GetSourceScoped`. A separate test with all-successful requests verifies `PartialSyncError` is nil and `SuccessCount == len(reqs)`.
Workload: 2.0 MM

### Task 2: Add StartedAt/CompletedAt to SyncEvent
Outcome IDs: outcome-3
Outcome Role: primary_product
Decommission IDs: decommission-4
Change Type: change
Description: The `SyncEvent` struct has `CreatedAt` and `RemoteRevision` but lacks `StartedAt` and `CompletedAt` time fields needed for sync diagnostics and doctor reports. Add these fields, populate them in `SyncToCache`, expose them on `SyncResult`, and define the incomplete-event interpretation for `sync_status`/doctor consumers.
Existing Behavior / Reuse: `SyncEvent` is defined in `cache/models.go:328` with fields `CreatedAt`, `RemoteRevision`, `Status`, `SourceID`, `RemoteType`, `RemoteID`, `IdempotencyKey`, `Message`. `SyncToCache` (service.go:753-838) records an in-progress event (line 792) with `CreatedAt` set to `s.now().UTC()` — this is the natural `StartedAt` point. The succeeded event (line 825) is the `CompletedAt` point. The `sync_events` table (cache/schema.go:173-186) has `created_at TEXT NOT NULL`. `SQLiteStore.RecordSyncEvent` (cache/accessors.go:434-450) inserts/reads sync event rows. `UpsertSyncGraph` writes full `SyncGraph` including its `SyncEvents` slice. `syncResultFromEvent` (service.go:1279-1286) populates `SyncResult` from a single `SyncEvent`. `GetSyncStatus` reads the latest `SyncEvent` per source.
Detailed Design: Add `StartedAt time.Time` and `CompletedAt time.Time` fields to `cache.SyncEvent` (cache/models.go). Add `started_at TEXT NOT NULL DEFAULT ''` and `completed_at TEXT NOT NULL DEFAULT ''` columns to `sync_events` CREATE TABLE (cache/schema.go). Update `SQLiteStore.RecordSyncEvent` INSERT/SELECT to include `started_at` and `completed_at` (cache/accessors.go). Update `UpsertSyncGraph`'s write path for sync events to include the new columns. In `SyncToCache` (service.go): set `inProgress.StartedAt = now` at the in-progress event creation point; at the succeeded event creation point, set `graph.SyncEvents[last].CompletedAt = now` and `graph.SyncEvents[last].StartedAt = inProgress.StartedAt`. Add `StartedAt time.Time` and `CompletedAt time.Time` to `SyncResult` (service/types.go), populated from the succeeded `SyncEvent` in the result-assembly block. Update `syncResultFromEvent` to copy `StartedAt` and `CompletedAt`. Add `LastSyncStartedAt time.Time` and `LastSyncCompletedAt time.Time` to `SyncStatusSummaryResult` (service/types.go). Update the method populating `SyncStatusSummaryResult` to select only events where `CompletedAt` is non-zero (completed events), then pick the event with the maximum `CompletedAt`; events where `StartedAt` is populated but `CompletedAt` is zero are classified as "incomplete" and are reportable by `sync_status` as in-progress syncs but are NOT considered for `LastSyncCompletedAt`.
Acceptance Criteria: A Go test `go test -run TestSyncEventTimestamps -count=1 ./internal/service/` constructs `*Service` with a mock store, calls `SyncToCache` with a fixture source, and verifies the returned `SyncResult.StartedAt` is non-zero, `SyncResult.CompletedAt` is non-zero, and `SyncResult.CompletedAt.After(SyncResult.StartedAt)`. A second test inserts two `SyncEvent` rows for the same source — one with `CompletedAt` zero (incomplete) and one with `CompletedAt` non-zero — then verifies `SyncStatusSummaryResult.LastSyncCompletedAt` corresponds to the row with the maximum non-zero `CompletedAt`, and the zero-`CompletedAt` row is excluded from the selection. A cache-level test verifies that a `SyncEvent` row inserted via `RecordSyncEvent` has non-empty `started_at` and `completed_at` columns when read back.
Workload: 2.0 MM

### Task 3: Add ZeroDelta to SyncResult with persistent event
Outcome IDs: outcome-3
Outcome Role: primary_product
Decommission IDs: decommission-4
Change Type: add
Description: `stageIssue` and `stageWiki` already compute `contentHash`, compare against existing cache records, and set `Skipped`/`Updated`/`Inserted` counts. A re-sync with no remote changes produces `Skipped == Fetched > 0` but no `ZeroDelta` signal exists on `SyncResult`. Add `ZeroDelta bool` set when all fetched content was skipped, and always persist a `SyncEvent` with `ZeroDelta: true` and full count metadata.
Existing Behavior / Reuse: `stageIssue` (service.go) computes `hash := contentHash(...)`, looks up existing source, and if `existing.ContentHash == hash` sets `counts.Skipped = 1`. `stageWiki` has identical pattern. `SyncCounts` (types.go) has `Fetched`, `Skipped`, `Updated`, `Conflicts`, `Inserted`. `SyncResult` has `Counts SyncCounts` and `Status`. `SyncToCache` returns `SyncResult` with populated counts. The counts correctly reflect delta — `Skipped` counts unchanged content.
Detailed Design: Add `ZeroDelta bool` field to `SyncResult` struct (service/types.go). In `SyncToCache`, after `fetchAndStage` returns successfully and `counts` is populated, compute: `ZeroDelta = (counts.Fetched > 0 && counts.Skipped == counts.Fetched && counts.Updated == 0 && counts.Inserted == 0 && counts.Conflicts == 0)`. Set `result.ZeroDelta = ZeroDelta`. ALWAYS record a succeeded `SyncEvent` via `UpsertSyncGraph` regardless of `ZeroDelta` — do not suppress the event. Add `ZeroDelta bool` field to `SyncEvent` struct (cache/models.go). When `ZeroDelta` is true, set `event.ZeroDelta = true` on the succeeded event while preserving all count fields. Add `zero_delta INTEGER NOT NULL DEFAULT 0` column to `sync_events` table (cache/schema.go). Update `SQLiteStore.RecordSyncEvent` to INSERT/SELECT `zero_delta`. Add `ZeroDelta bool` to `SyncStatusSummaryResult` (service/types.go).
Acceptance Criteria: A Go test `go test -run TestZeroDeltaPersistentEvent -count=1 ./internal/service/` calls `SyncToCache` twice with unchanging fixture data; the second call returns `SyncResult.ZeroDelta == true`, `SyncResult.Counts.Fetched > 0`, `SyncResult.Counts.Skipped == SyncResult.Counts.Fetched`; a store query verifies two `SyncEvent` rows exist for that source (both in-progress and succeeded persisted), the second succeeded event has `ZeroDelta == true`, and count fields are preserved. A separate test with content that actually changed between syncs verifies `ZeroDelta == false`. The `sync_status` summary for the zero-delta source reflects `ZeroDelta: true` from `SyncStatusSummaryResult`.
Workload: 1.5 MM

## Cross-Cutting Constraints
- All sync output must pass through redaction filter before print/log — sync engine returns `error` values and `SyncResult` structs; redaction is applied by the CLI layer in `writeError`/`renderSyncText`
- The sync engine must not open or migrate the cache database itself — it receives an already-opened `cache.Store` from the service factory that has passed schema version validation
- The existing `fetchAndStage` retry loop (service.go:1026-1059) is retained. It retries on `ErrRateLimited` and `ErrNetworkUnavailable` with configurable attempts via `SyncRequest.MaxAttempts`. When `SyncResources` encounters a resource that exhausts retries, the resulting error (provider-level or network-level) is captured as a `ResourceError` in `PartialSyncError.Errors` — the `PartialSyncError` does not distinguish retry-exhaustion from first-attempt API errors at the error-type level; the wrapped `Err` preserves the underlying cause for diagnostic rendering.
- `ZeroDelta` events must always be persisted — suppression would lose the content hash basis, `RemoteRevision`, and durable evidence that re-sync occurred

## Data And Control Flow
- CLI `sync` command resolves provider mode from `--live` flag → constructs `*Service` via `NewWithClient` with live `gitcode.Client` or `New` with fixture default → calls `svc.SyncResources(ctx, reqs)` for multi-resource sync or `svc.SyncToCache(ctx, req)` for single → `SyncToCache` resolves idempotency key, records in-progress event with `StartedAt` → calls `fetchAndStage` → `fetchAndStage` retry loop calls `fetchOnce` → `fetchOnce` dispatches to `s.client.GetIssue` or `s.client.GetWikiPage` via `gitcode.Client` → `stageIssue`/`stageWiki` computes `contentHash`, compares with cache, sets counts → `SyncToCache` computes `ZeroDelta`, appends succeeded event with `CompletedAt` and `ZeroDelta` flag ALWAYS → persists via `UpsertSyncGraph` → returns single result to `SyncResources` → `SyncResources` aggregates into `SyncResourcesResult` with `SuccessCount`/`FailureCount` and optional `PartialSyncError` → CLI renders result
- `sync_status` CLI command reads latest `SyncEvent` per source from cache → selects only events with non-zero `CompletedAt`, then picks event with max `CompletedAt` → reports `LastSyncStartedAt`, `LastSyncCompletedAt`, `ZeroDelta`, `RemoteRevision`, freshness. Events with `StartedAt` populated but `CompletedAt` zero are classified as "incomplete" and may be reported separately as in-progress syncs, but are excluded from `LastSyncCompletedAt` selection. Doctor command consumes `SyncStatusSummaryResult` for its sync section

## Component Interactions
- `cmd/gitcode-mcp/` → `internal/service/` — CLI constructs `*Service` via `service.NewWithClient` with live `gitcode.Client` when `--live` is set, or `service.New` for fixture default; calls `SyncResources` for multi-resource sync or `SyncToCache` directly; consumes `SyncResourcesResult`, `SyncResult`, `error`, and `PartialSyncError`; `renderSyncText` and `renderSyncStatusSummaryText` format user-facing output from sync-engine struct fields
- `internal/service/` → `internal/gitcode/` — Sync engine calls `s.client.GetIssue`, `s.client.GetWikiPage`, `s.client.ListIssueComments` through the `gitcode.Client` interface; does not depend on concrete client type
- `internal/service/` → `internal/cache/` — Sync engine writes source graphs and sync events via `store.UpsertSyncGraph`, records individual events via `store.RecordSyncEvent`, reads sources via `store.GetSourceScoped`, checks idempotency via `store.GetSyncEventByKey`; cache owns schema versioning and migration
- `internal/service/` → `cmd/gitcode-mcp/` — CLI sync command consumes `SyncResourcesResult`, `SyncResult`, `error`, and `PartialSyncError` from the sync engine

## Rationale
The sync engine in `internal/service/` is already well-architected for provider dispatch (`*Service.client` field supporting any `gitcode.Client`), hash-based delta detection (`contentHash` comparison in `stageIssue`/`stageWiki`), retry (`fetchAndStage` loop), and sync event recording (`UpsertSyncGraph`). The three tasks add genuinely missing capabilities: multi-resource partial-failure semantics via `SyncResources` + `PartialSyncError` + `SyncResourcesResult` (required by architecture but not implemented), `StartedAt`/`CompletedAt` timestamps on `SyncEvent` (neither field exists), and `ZeroDelta` detection with persistent events (counts are correct but the signal is not captured). The component ID discrepancy (`internal/sync/` in architecture vs `internal/service/` in source) is explicitly noted; all tasks anchor to `internal/service/` types where the implementation actually lives.
